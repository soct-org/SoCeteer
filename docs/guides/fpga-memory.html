FPGA Memory & Custom DDR4

How SoCeteer decides how much DRAM your design can address, how to select the DIMM that is actually inserted in the board with --ext-mem-part, and what happens under the hood when that DIMM differs from the board preset.

The memory part is the single source of truth

The Vivado DDR4 IP sizes its address decode window from the selected memory part (the DIMM's Vivado catalog name, e.g. MTA8ATF51264HZ-2G1). SoCeteer derives everything else from that part, so the ExtMem configuration, the generated device tree and the controller's decode window stay consistent by construction:

  • the capacity (via the PartRegistry),
  • the ExtMem size seen by RocketChip,
  • the memory node in the DTB (what programs and mem-test see),
  • the block design's address aperture.

A mismatch between the selected part and the physical DIMM silently truncates usable memory to the smaller of the two — which is why the part is a launcher argument.

Selecting a DIMM: --ext-mem-part

sbt "runMain soct.SOCTLauncher --target vivado --board ZCU104 \
  --ext-mem-part MTA16ATF2G64HZ-2G3 --vivado /path/to/vivado"
  • Pass it once per external DDR4 port (first occurrence configures the first port, and so on).
  • If omitted, the board's preset part is used (ZCU104: MTA8ATF51264HZ-2G1, 4 GiB).
  • Physical module labels often carry a die-revision suffix (MTA16ATF2G64HZ-2G3H1); SoCeteer strips it automatically, since the Vivado catalog lists parts without it.

The part registry

Capacity resolution (see PartRegistry) works in two steps:

  1. Exact match against the registry (base part name without the speed grade).
  2. Micron pattern fallback: MTA<n>ATF<depth><width>… encodes the organisation, e.g. MTA8ATF2G64HZ = 2G locations × 64 bit = 16 GiB (ECC x72 check bits are not addressable and don't count).

Parts from other vendors or with non-standard names must be added to the registry — the launcher fails loudly at argument-validation time if a part's capacity cannot be derived.

Board flow vs. custom interface

Vivado's board flow configures the DDR4 IP from the board preset — convenient, but the preset locks C0.DDR4_MemoryPart (the parameter is disabled; set_property on it is silently ignored). The board's preset DIMM is therefore the hard limit of what board-flow designs can address, regardless of what is physically inserted.

When the requested part differs from the preset, SoCeteer switches that port to a custom (non board-flow) interface:

  • The DDR4 IP is configured explicitly (memory part, type, data width, clock timing) instead of via the board association. The IP derives all part-dependent timing (CAS latency, rank count) from the part itself.
  • Pin locations come from the board definition's ddr4PinMap (taken from the board files' part0_pins.xml, including second-rank pins the single-rank preset never uses) and are emitted as a generated XDC.
  • The interface port is created by externalizing the configured controller pin (make_bd_intf_pins_external) instead of create_bd_intf_port, so the port inherits DIMM-dependent signal widths — a dual-rank module widens cs_n/cke/odt/ck.
  • The board associations placed three things that the custom interface must now place itself: the DDR4 pins, the differential system clock pair, and the board reset pin. All three are emitted into the same XDC; missing board data fails at generation time, not hours into implementation.

Multi-channel memory and the address deinterleaver

With multiple memory channels, RocketChip distributes cache lines round-robin across the channels: each channel's exported address sets pin the channel-select bits, so the addresses arriving at one channel are sparse — every other cache line is a hole. A DDR controller fed these raw addresses would waste 1−1/N of its capacity.

SoCeteer inserts an AXIAddrDeinterleaver per channel (a purely combinational AXI4 pass-through module) that drops the constant channel-select bits and compacts the sparse view onto the dense range [base, base + capacity). The interleave geometry is derived automatically from the diplomacy address sets; contiguous (single-channel) designs get no deinterleaver.

Validating on hardware

Two bundled programs verify the result end to end:

  • mem-test — reads the memory range from the DTB, then runs a capacity probe (detects aliasing/wrap-around from a mis-sized decode window), a cache-line-edge test, and moving-inversion sweeps over the full range.
  • heap-test — allocates the DTB-bounded heap and probes the newlib allocator limits.
