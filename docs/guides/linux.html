Documentation / Booting Linux

Booting Linux

The binaries/linux project — a separate CMake tree with its own LLVM/musl toolchain — builds everything that involves the Linux kernel: bootable images (an OpenSBI fw_payload wrapping the kernel, the design's device tree and an embedded initramfs), userspace programs, and out-of-tree kernel modules. It is configured against the same SOCT_SYSTEM file as the bare-metal project (see the Binaries guide for the system file, building and JTAG flashing), with kernel and OpenSBI source trees dropped in as binaries/linux/linux-stable/ and binaries/linux/opensbi/. A boot image is copied to the SD card as BOOT.ELF — the name the sd-boot boot ROM loads — and boots like any other program.

The device tree's journey: boot ROM, OpenSBI, Linux

For the Linux boot image, the device tree is not consulted once — it flows down the whole boot chain, and each stage uses and reshapes the same tree. Nothing is ever literally merged from two trees; one blob is passed along and amended:

  1. Boot ROM. The generator's DTB is baked into the boot ROM, and sd-boot hands its address to whatever it loads in register a1 (hart id in a0) — the RISC-V boot convention every later stage speaks as well. Bare-metal programs consume this ROM copy directly (see Bare-Metal Programs & soctglue).
  2. BOOT.ELF ignores the ROM copy. The boot image embeds its own DTB, compiled from the design's DTS at build time (OpenSBI's FW_FDT_PATH) with slack space added (dtc -p). Two reasons: the ROM copy is not writable, and the next stage must write to the tree; and the boot image becomes self-contained — it does not care which ROM DTB the flashed bitstream carries, only that the embedded one matches the flashed hardware.
  3. OpenSBI reads it… The firmware's own drivers are device-tree probed exactly like Linux's: the CLINT it uses for timers and IPIs and the UART behind its banner output come from the same nodes Linux will bind later — one description serving two consumers.
  4. …then relocates and fixes it up. OpenSBI copies the tree to a configured DRAM address (FW_PAYLOAD_FDT_ADDR) and edits it in place: it inserts reserved-memory nodes covering its own footprint — the memory it shields with PMP so the OS can never touch machine-mode firmware. These appear in a boot log as mmode_resv* regions. This is the closest thing to a “merge” in the chain: the hardware description gains firmware-owned annotations before the OS ever sees it.
  5. Linux receives the fixed-up tree — a1 again, at the S-mode kernel entry. The kernel parses the memory node into its zones (honoring the reservations from step 4), binds drivers by compatible, and reads device properties (the UART's current-speed, without which the console driver refuses to probe). The command line, too, comes from the tree: the generator emits a /chosen node whose bootargs name the design's UART as console and early console — hardware facts, kept with the hardware description. (A kernel can also carry a built-in CONFIG_CMDLINE; by default that is only a fallback for trees without /chosen, and this build leaves it unset so the tree is the single source.)

The corollary: the embedded DTB is a snapshot of one hardware generation. Rebuild the boot image whenever the design changes — a tree describing more memory than the flashed decode backs, or devices at moved addresses, fails at run time, not loudly at build time.

Userspace programs

Programs that run under the Linux kernel instead of on bare metal follow the same directory-plus-template pattern as the rest of binaries/, with two templates:

cmake_minimum_required(VERSION 3.21)

list(APPEND CMAKE_C_SRCS ${CMAKE_CURRENT_SOURCE_DIR}/main.c)

include(${CMAKE_CURRENT_LIST_DIR}/../userspace.cmake)  # ELF only
# ...or ../initram.cmake to also get a bootable image (below)

The target (named after the directory) is a real add_executable — IDEs index it with full cross-compilation flags — producing a static riscv64-linux-musl ELF, copied to SOCT_ELFS_DIR. Static, because the target system is initramfs-only; there is no shared-library ecosystem to link against. soctglue, the bare-metal linker script and the syscall handler chain play no role here: these are ordinary Linux programs with a full C library. A program that includes initram.cmake instead additionally gets a <name>-boot-elf target: a complete <name>.BOOT.ELF with the program as the initramfs /init — running a userspace program on hardware becomes build, copy to SD, boot. The console-echo bring-up demo (linux/userspace/echo, which must never exit — it is PID 1) is the reference example. A boot image need not wrap a compiled program at all: linux/userspace/shell compiles nothing and instead hands its own initramfs description to soct_linux_add_boot_image(<name> DESC <desc> DEPENDS <packed files…>), packing the prebuilt BusyBox and a boot script (see Boot images below). For IDE support, open binaries/linux as its own CMake project configured with the same -DSOCT_SYSTEM.

Boot images

The project's targets:

  • linux-sysroot — the riscv64-linux-musl sysroot (built automatically at configure time when missing, shared across build trees under shared/vendor/); see Userspace programs above.
  • boot-dtb — the design DTS compiled with fixup padding; it is embedded into the firmware, so boot images do not depend on the ROM copy of the DTB (but must match the flashed hardware).
  • <name>-boot-elf — for every program that includes initram.cmake: <name>.BOOT.ELF in SOCT_ELFS_DIR, a kernel with that program as /init. The kernel and OpenSBI build directories are shared: the ~2000 kernel objects compile once, and switching programs re-runs only the initramfs packing and the links (the initramfs description is a fixed path in the kernel build dir that each image overwrites — never a reconfigure). Image builds are serialized through a job pool; honored by the Ninja generator. With the SOCT_FLASH_* variables set, each image also gets a <name>-flash target that JTAG-loads it (see Flashing to an FPGA) — booting Linux without touching the SD card.
  • <name>-driver — for every Kbuild directory under linux/drivers/: an out-of-tree kernel module, built with kbuild against the shared kernel build (matching vermagic by construction). Boot images pack the .ko files into their initramfs, /init loads them, and the kernel's initramfs rules re-pack a rebuilt module automatically.

The kernel configuration is the riscv defconfig plus linux/linux.fragment.in: the UART Lite console driver, the Xilinx interrupt-controller and DMA-engine drivers, a pinned HZ, the initramfs, and AppArmor disabled (useless on this SoC, and its header generation is the one build step requiring GNU sed's \L on every host) — why the SoC needs exactly these drivers is the subject of SoC-specific behavior below. The command line is deliberately not baked into the kernel: the generator emits a /chosen node whose bootargs select the design's UART as console and early console, so one kernel image serves any hardware generation.

The shell image (shell.BOOT.ELF) boots into an interactive BusyBox shell. Its /init (linux/userspace/shell/init.sh) mounts the pseudo filesystems, installs the applet links and keeps a job-controlled shell alive on the console (setsid + cttyhack; the script itself stays PID 1, so an exiting shell is respawned instead of panicking the kernel). PID 1 also answers shutdown requests: reboot without -f never calls the kernel — it signals init — so the script traps SIGTERM into the real reboot -f syscall, which lands in the SBI SRST reset (see reboot below); halt/poweroff report that the FPGA has no software power control. BusyBox is not built from source — that would add a second kbuild frontier for one binary — but taken from Debian's busybox-static riscv64 package: a static rv64gc binary, hash-pinned and fetched once at configure time into shared/vendor/busybox/ from snapshot.debian.org, whose archive URLs are permanent (regular pool URLs disappear once a package version is superseded).

SoC-specific behavior

Everything a Linux user notices that is particular to this SoC, and how each piece is achieved — from the syscall down to the block design.

Console and interrupts (ttyUL0)

The console is the design's UART Lite. The generated device tree makes it work without any configuration: the /chosen bootargs select it as console and early console, and the node carries the current-speed property without which the kernel driver refuses to probe. Behind it sits a deliberate piece of block-design topology: the UART Lite signals its interrupt as a one-clock pulse, and the PLIC's per-source gateways can miss a pulse that arrives while a previous claim is still in flight — after which that source would stay silent forever (the original symptom: a console that froze under load, permanently). All fabric interrupts therefore cascade through an AXI INTC: the UART lands on an edge-latching input, SDC and VDMA on level inputs, and the INTC's single level-typed output is a PLIC source that can never be missed. Linux binds this with its Xilinx interrupt-controller driver as a chained irqchip — /proc/interrupts shows the peripherals under the INTC domain, not the PLIC directly. One kernel patch (linux/patches/, an upstream candidate) makes that binding order-proof: mainline registers the driver only for early-boot probing, which orphans an INTC whose parent is the later-probing PLIC platform driver; the patch adds a platform-driver registration with probe deferral.

Storage (/dev/mmcblk0)

The SD card the system booted from reappears inside Linux as a block device. The SD controller has no mainline driver; the out-of-tree module in linux/drivers/sdc provides one, and the shell image's initramfs carries and loads it at boot — its partitions then mount normally from the shell. Card detect is a debounced presence level straight from the RTL, with the PMOD's active-low CD line inverted in the hardware configuration — so a card already inserted at power-on enumerates during boot, and hot insert/removal both work. The root filesystem stays the initramfs embedded in the kernel image until the dynamic-userspace world (linux/rootfs) exists; the card is data storage.

reboot

A working reboot crosses every layer of the stack; each one contributes a small piece:

  1. Userspace: BusyBox reboot does not call the kernel — it signals PID 1, and the shell image's /init traps that into the real reboot(2) syscall (reboot -f skips straight to the syscall).
  2. Kernel: no reset driver of its own — the RISC-V kernel forwards the request to the firmware through the SBI SRST extension.
  3. OpenSBI: its generic platform bound a syscon-reboot reset backend at boot, from two generated device-tree nodes: a syscon regmap covering the system-reset register and a reboot node naming offset, value and mask. The SRST call becomes a single M-mode store of 1 to that register.
  4. Block design: the register is an AXI GPIO output, OR-wired (sw_or_hw_reset) with the hardware reset sources — the board reset and, when a debug module is present, its ndreset — into the ext_reset_in of the core and periphery reset synchronizers: the same net the JTAG reset_core flow pulses.
  5. Back to the beginning: core and periphery restart while the DDR domain is untouched and keeps its calibration; the boot ROM runs again and reloads BOOT.ELF from the card. The register clears itself — the reset it raises also resets the GPIO driving it — so the system comes back up instead of boot-looping.

halt/poweroff: the FPGA has no software power control, and /init says so instead of pretending.

Video (VDMA)

In designs with the video stream, the generated VDMA node binds the kernel's Xilinx DMA-engine driver as a dmaengine provider — channel, frame stores and data width all described by the device tree. No framebuffer driver sits on top of it yet: the DisplayPort pipeline is exercised by the bare-metal dp-test (see DisplayPort Video), and Linux framebuffer glue is future work.

Toolchains

Two toolchains split the work, deliberately:

  • LLVM/Clang builds the kernel and OpenSBI, through their upstream LLVM=<bindir>/ build mode. The vendored bare-metal binutils cannot link either one: their ld has no dynamic-linking support, OpenSBI links with -pie unconditionally, and a riscv MMU kernel must link its vDSO as a shared object (the failure shows up indirectly, as missing __vdso_*_offset symbols).
  • The repo's vendored toolchain builds everything bare-metal — programs and boot ROMs — and contributes one artifact to the Linux world: its rv64/lp64d libgcc.a, whose OS-agnostic builtins (soft fp128 arithmetic and friends) stand in for the compiler-rt that prebuilt LLVM does not ship for cross targets.

The LLVM toolchain is auto-detected by FindLLVMRiscv: candidates come from PATH, Homebrew (including its split llvm/lld keg-only formulas, for which a unified directory of symlinks is assembled), Debian's /usr/lib/llvm-* and MacPorts — and each candidate is proved by linking a RISC-V PIE through its ld.lld rather than trusted by name or version. That probe is why Apple's Xcode clang never qualifies: it ships without the RISC-V backend and without ld.lld. GNU make is found the same way (FindGNUMake: every gmake/gnumake/make candidate is version-checked, so an ancient system make cannot shadow a usable one). Userspace programs get their cross flags from the project's toolchain file (shared/cmake/toolchain-riscv-linux-musl.cmake), built on the same discovery. Any unmet requirement fails the binaries/linux configure loudly, with the remedy in the message; the bare-metal project is untouched by construction — the two worlds are separate build trees.

Host requirements

Every host:

  • LLVM with the RISC-V backend plus ld.lld and the llvm-* binutils (one directory or Homebrew's split installation).
  • GNU make ≥ 4.0 (the kernel refuses older), plus host flex, bison and bc; dtc for the device tree.
  • A build directory path without commas or spaces: OpenSBI passes linker scripts as -Wl,-T<path> and the compiler driver splits -Wl arguments at every comma, while GNU make cannot handle spaces in paths at all. Such paths are refused at configure time (IDE-generated build directories named after profiles are the usual offender).

Linux hosts: the distribution packages are sufficient — on Debian/Ubuntu: clang lld make flex bison bc device-tree-compiler. The repo's Docker image ships all of them; no accommodations of any kind are applied on Linux.

macOS (Darwin) hosts: four Homebrew packages — make, llvm, lld (a separate formula; llvm alone ships no linker) and gnu-sed. Beyond that, the build applies a set of accommodations automatically, all scoped to the kernel's host tools on Darwin only — target code and Linux hosts never see any of them:

AccommodationWhy
linux/host-compat/elf.h (verbatim musl, MIT) on the host include path macOS ships no <elf.h>; the kernel's host tools (scripts/elf-parse.h users, modpost) require one.
linux/host-compat/byteswap.h glibc-only header (bswap_*) used by modpost; shimmed with compiler builtins.
linux/host-compat/darwin-compat.h, force-included usr/gen_init_cpio.c uses O_LARGEFILE and copy_file_range, which Darwin lacks; defined to 0 and a read/write fallback respectively.
-D_UUID_T -D__GETHOSTUUID_H The macOS SDK typedefs uuid_t as a bare byte array (pulled in via unistd.h), colliding with the kernel's struct definition in scripts/mod/file2alias.c; the defines suppress the SDK's.
PATH shim mapping sed → gsed for kernel invocations BSD sed renders a replacement \t as a literal t, mangling every macro in the generated vDSO offsets header (__vdso_…_offsett0x…) into nonsense.
OpenSBI build directory pre-created BSD readlink -f fails on nonexistent paths, which derails OpenSBI's O= handling before the build starts.
