Simulation & Debugging

The Verilator target turns the emitted Verilog into a native simulator binary that loads RISC-V ELFs, bridges their syscalls to the host, and exposes a JTAG interface for live debugging with OpenOCD/GDB.

Background

How simulating an SoC works at all. Readers who have used Verilator and FESVR can skip ahead.

RTL simulation with Verilator

Verilator does not interpret Verilog; it compiles the design into C++ that evaluates the whole circuit once per clock cycle, then links it with a user-provided C++ harness into a native executable. The result is a cycle-accurate model of the SoC — every register behaves exactly as in hardware, only some orders of magnitude slower (and with simplifications: two-state logic, no analog effects). The point is observability and turnaround: any signal can be inspected on any cycle, and a design change is a recompile away instead of an hours-long place-and-route.

The host bridge: DPI-C, HTIF and FESVR

A simulated SoC has no monitor, SD card or serial port — but it runs inside a host process that can stand in for all of them. SystemVerilog's DPI-C interface lets simulated hardware call C functions, and over this bridge runs HTIF (Host-Target Interface), the Berkeley protocol through which the target reaches its host. FESVR (the frontend server) is the host side of it: it loads the ELF, services the target's proxied syscalls and forwards its exit code. The net effect — a bare-metal RISC-V program behaving like an ordinary command-line tool — is what makes the workflow below possible.

Waveforms

A simulator can record every signal transition into a waveform file (VCD being the classic format) for post-mortem inspection in a viewer like GTKWave — the hardware equivalent of an execution trace. Recording everything is expensive, which is why tracing is a compile-time opt-in with a depth limit rather than a default.

Debugging: the same stack as hardware

RISC-V defines a standard debug module: a piece of hardware in the SoC that can halt harts, read registers and access memory, controlled over JTAG. GDB does not speak JTAG; OpenOCD translates between GDB's remote protocol and the debug transport. In simulation the JTAG signals are exposed over a TCP socket (OpenOCD's remote bitbang protocol) instead of a cable — everything above that socket is identical to debugging the FPGA, which is exactly why the workflows transfer.

How the simulator is built

The sim/ CMake project verilates the emitted design and compiles it together with a C++ harness (FESVR, the DPI-C bridge, a disassembler for tracing) into one binary. It finds the design through the SOCT_SYSTEM variable — the SOCTSystem.cmake file every launcher run emits — or falls back to the SOCTSystem-latest.cmake symlink at the project root. The quick-start commands are in the README.

Build-time knobs worth knowing (see sim/CMakeLists.txt for the rest): -DVL_TRACE=1 compiles in VCD waveform tracing (VL_TRACE_DEPTH limits the traced hierarchy, default 2 — tracing slows the simulation considerably, so it's off by default), and VL_THREADS overrides the Verilator thread count (default: the simulated SoC's CPU count + 1, not the host's core count).

Running programs

The simulator takes the ELF as its first positional argument; everything else is optional and listed by --help (log level/file, reset cycles, VCD output path, ...).

build/sim-build/simulator path/to/program.elf --log-level=debug

Syscalls made by the program (printf, file I/O, exit) travel over the HTIF interface to the host via FESVR, so bare-metal programs behave like command-line tools: stdout appears on your terminal and the program's exit code becomes the simulator's. argc/argv are forwarded to the program's main.

Debugging with OpenOCD and GDB

The simulated SoC's debug module is reachable over a JTAG remote bitbang socket (port 1337). A matching OpenOCD config ships in sim/remote/rocket.cfg:

# Terminal 1: run the simulation
build/sim-build/simulator program.elf

# Terminal 2: attach OpenOCD to the simulated JTAG
openocd -f sim/remote/rocket.cfg

# Terminal 3: debug like real hardware
riscv-none-elf-gdb program.elf -ex "target extended-remote :3333"

This is the same debug path as on the FPGA (where the debug module sits behind the BSCAN bridge instead), so GDB workflows transfer 1:1 between simulation and hardware.

Regression testing

The scalatest suite (SimulationSpec) runs the full pipeline — emit, build, simulate a syscall test — for every entry of the test matrices, which are also the single source of truth for the CI matrix. Add a config/xlen pair to the matrix JSON and both CI and the local suite pick it up.
