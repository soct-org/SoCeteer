Simulation & Debugging

The Verilator target turns the emitted Verilog into a native simulator binary that loads RISC-V ELFs, bridges their syscalls to the host, and exposes a JTAG interface for live debugging with OpenOCD/GDB.

How the simulator is built

The sim/ CMake project verilates the emitted design and compiles it together with a C++ harness (FESVR, the DPI-C bridge, a disassembler for tracing) into one binary. It finds the design through the SOCT_SYSTEM variable — the SOCTSystem.cmake file every launcher run emits — or falls back to the SOCTSystem-latest.cmake symlink at the project root. The quick-start commands are in the README.

Build-time knobs worth knowing (see sim/CMakeLists.txt for the rest): -DVL_TRACE=1 compiles in VCD waveform tracing (VL_TRACE_DEPTH limits the traced hierarchy, default 2 — tracing slows the simulation considerably, so it's off by default), and VL_THREADS overrides the Verilator thread count (default: core count + 1).

Running programs

The simulator takes the ELF as its first positional argument; everything else is optional and listed by --help (log level/file, reset cycles, VCD output path, ...).

build/sim-build/simulator path/to/program.elf --log-level=debug

Syscalls made by the program (printf, file I/O, exit) travel over the HTIF interface to the host via FESVR, so bare-metal programs behave like command-line tools: stdout appears on your terminal and the program's exit code becomes the simulator's. argc/argv are forwarded to the program's main.

Debugging with OpenOCD and GDB

The simulated SoC's debug module is reachable over a JTAG remote bitbang socket (port 1337). A matching OpenOCD config ships in sim/remote/rocket.cfg:

# Terminal 1: run the simulation
build/sim-build/simulator program.elf

# Terminal 2: attach OpenOCD to the simulated JTAG
openocd -f sim/remote/rocket.cfg

# Terminal 3: debug like real hardware
riscv-none-elf-gdb program.elf -ex "target extended-remote :3333"

This is the same debug path as on the FPGA (where the debug module sits behind the BSCAN bridge instead), so GDB workflows transfer 1:1 between simulation and hardware.

Regression testing

The scalatest suite (SimulationSpec) runs the full pipeline — emit, build, simulate a syscall test — for every entry of the test matrices, which are also the single source of truth for the CI matrix. Add a config/xlen pair to the matrix JSON and both CI and the local suite pick it up.
