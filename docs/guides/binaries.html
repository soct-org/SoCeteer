Binaries

Everything that runs on the SoCeteer-generated SoC lives under binaries/, organized by execution context. This guide covers what the contexts are and the infrastructure they share: the system file, building, and JTAG flashing. The two specialized guides build on it: Bare-Metal Programs & soctglue for programs without an OS and boot ROMs, and Booting Linux for the kernel, boot images, drivers and Linux userspace.

What runs where

Every directory is named after the execution context of the binaries it holds. Two independent CMake projects live side by side, because CMake allows one toolchain per build tree and the two worlds need different ones:

DirectoryExecution contextToolchain / runtimeDelivered by
bootroms/<name>/On-chip ROM, at reset (first stage) vendored bare-metal GCC, self-contained (no soctglue)baked into the bitstream
baremetal/<name>/Directly on the SoC, no OS (M-mode) vendored bare-metal GCC + soctglue/newlibsd-boot from SD, or JTAG (<name>-flash)
linux/userspace/<name>/Linux userspace, PID 1 or initramfs tool LLVM/Clang + static musl (own project)inside a boot image (<name>.BOOT.ELF), flashable
linux/drivers/<name>/Linux kernel (out-of-tree modules) kernel kbuild against the shared build dir.ko inside a boot image's initramfs, loaded by /init
linux/rootfs/Linux userspace, dynamically linked — reserved LLVM/Clang + shared muslsee its README

Programs and boot ROMs are auto-discovered subdirectories. Each directory's CMakeLists includes one of the shared templates — and which template it includes decides what kind of binary it becomes:

TemplateProjectProduces
baremetal/soctglue-static.cmakebinaries/ A bare-metal ELF linked against soctglue and newlib, laid out by soct.ld, flashable over JTAG
bootroms/template.cmakebinaries/ A self-contained boot ROM, stripped to the raw image the bitstream embeds
linux/userspace/userspace.cmakebinaries/linux/ A static riscv64-linux-musl ELF (a real add_executable)
linux/userspace/initram.cmakebinaries/linux/ The same, plus a bootable <name>.BOOT.ELF with the program as the initramfs /init

Directory layout

binaries/                          # The bare-metal CMake project (vendored riscv toolchain)
  bootroms/<name>/                 # Runs from on-chip ROM at reset (first stage)
  bootroms/template.cmake          # Template for boot ROMs (self-contained, no soctglue)
  baremetal/<name>/                # Runs directly on the SoC, no OS (M-mode, soctglue)
  baremetal/soctglue-static.cmake  # Template: bare-metal programs (soctglue + newlib)
  baremetal/xsdb-flash.cmake       # Included by the bare-metal template: JTAG flash targets
  linux/                           # A SEPARATE CMake project: everything Linux
    userspace/<name>/              # Runs in Linux userspace (real add_executable targets)
    userspace/userspace.cmake      # Template: static riscv64-linux-musl ELF
    userspace/initram.cmake        # Template: userspace + a bootable <name>.BOOT.ELF
    linux-stable/, opensbi/        # Drop-in kernel and OpenSBI source checkouts
    drivers/<name>/                # Out-of-tree kernel modules (auto-discovered Kbuild dirs)
    patches/                       # Minimal upstream-candidate kernel patches (see README)
    rootfs/                        # Reserved: dynamic userspace (see README)
shared/soctglue/                   # Runtime library: soct.ld, crt0, syscall layer, FatFs glue

binaries/linux is its own project because CMake allows one toolchain per build tree, and the bare-metal toolchain cannot target Linux — see Booting Linux.

Browse binaries/baremetal for the bundled examples and hardware-validation tools.

The system file (SOCT_SYSTEM)

Every build requires a system file — a CMake file generated by SoCeteer that describes the target SoC. It provides variables such as:

VariableMeaning
SOCETEER_ROOTAbsolute path to the SoCeteer checkout
SOCETEER_VERSIONVersion string checked by soctglue
SOCT_ARCH / SOCT_ABIRISC-V march/mabi flags (e.g. rv64gc / lp64d)
SOCT_TARGETBuild target (verilator, fpga, …)
SOCT_COMPILE_DEFSExtra compiler definitions from the SoC config
SOCT_DTS / SOCT_DTBPath to the device-tree source/blob
SOCT_ELFS_DIROutput directory for ELF files
SOCT_BOOTROM_IMGOutput path for the boot ROM binary image
SOCT_NCPUSNumber of CPU cores in the SoC
SOCT_CLINT_BASEBase address of the CLINT

The top-level CMakeLists.txt finds this file via FindSOCTSystem.cmake. It looks first at the SOCT_SYSTEM variable (passed with -DSOCT_SYSTEM=...), then falls back to a SOCTSystem-latest.cmake symlink in the repository root that SoCeteer creates automatically.

Building

cmake -S binaries -B build \
  -DSOCT_SYSTEM=/path/to/SOCTSystem.cmake \
  -DCMAKE_TOOLCHAIN_FILE=shared/cmake/toolchain-riscv.cmake
cmake --build build -j$(nproc)

ELF files land in SOCT_ELFS_DIR (set by the system file). The binaries/linux project configures the same way — a separate build tree against the same -DSOCT_SYSTEM, no toolchain file (it discovers its own toolchains; see Booting Linux for host requirements and targets).

Flashing to an FPGA

The bare-metal template ends by including xsdb-flash.cmake, so every bare-metal program gets a <name>-flash target that loads the compiled ELF onto the FPGA over JTAG using Xilinx xsdb — locally, or through an SSH host on which xsdb runs. The targets are only created when at least one of SOCT_FLASH_XSDB or SOCT_FLASH_HOST is set. The machinery is shared with the linux project (shared/cmake/SoctXsdbFlash.cmake), where the flashable artifact is the boot image: an initram.cmake program's <name>-flash target JTAG-loads its <name>.BOOT.ELF — a plain physical-address ELF — booting Linux with that program as /init without touching the SD card. A plain userspace ELF is still not flashable: JTAG writes physical memory with the hart halted and cannot deliver a file into a running kernel — such programs reach the target inside a boot image's initramfs.

cmake --build build --target hello-hart-flash

CMake variables

VariableDefaultDescription
SOCT_FLASH_XSDB(empty)Path to the xsdb binary (local, or on the remote host)
SOCT_FLASH_HOST(empty)SSH host to flash through — leave empty to flash locally
SOCT_FLASH_REMOTE_DIR/tmpDirectory on the remote host to upload the ELF into
SOCT_FLASH_BOOT_HART0Boot hart index written into register a0 before con

The DTB address written into register a1 is not a knob: it is the system file's SOCT_DTB_ADDR — the address the shared boot-ROM linker script pins the ROM's embedded device tree to, so the JTAG handoff matches what the boot ROM itself would pass.

Pass any of these at configure time:

cmake -S binaries -B build \
  -DSOCT_SYSTEM=/path/to/SOCTSystem.cmake \
  -DSOCT_FLASH_XSDB=/opt/Xilinx/Vivado/2024.2/bin/xsdb \
  -DSOCT_FLASH_BOOT_HART=0

Flash preludes (SOCT_FLASH_PRELUDE)

A program that needs extra steps before its ELF is loaded — board or subsystem initialization, staging auxiliary files — can set SOCT_FLASH_PRELUDE to a block of Python source before including the bare-metal template. It is injected into the generated flash wrapper and runs with elf, xsdb, host and remote_dir in scope, so one prelude serves local and remote flashing alike. The reference use is dp-test, whose prelude runs the Vivado-generated psu_init sequence the DisplayPort pipeline needs once per power-up.
