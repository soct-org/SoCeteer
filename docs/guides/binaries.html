Bare-Metal Programs & soctglue

The binaries/ directory contains RISC-V bare-metal programs and boot ROMs that run on the SoCeteer-generated SoC. All binaries share a common build infrastructure provided by soctglue.

Background

What “bare metal” actually implies. Readers who have written for microcontrollers can skip ahead.

No operating system means no environment

A bare-metal program is the only software on the machine. Nothing exists before its first instruction: no stack, no zeroed variables, no console, no files, no argv — the CPU simply starts executing at an address. Everything a hosted C program takes for granted must be established by the program itself, which is the job of the startup code (conventionally named crt0, “C runtime zero”): set up a stack, zero the BSS segment (the region holding zero-initialized globals, which only exists as a size in the binary), initialize globals, and only then call main().

The linker script decides where things live

With no loader and no virtual memory, the binary's layout is the memory layout: a linker script assigns each section (code, data, BSS) a physical address and defines symbols (like the end of the image) that the runtime uses to know where free memory begins. Stack and heap are not provided by anyone — they are just address ranges the linker script and runtime agree on, which is why this guide talks about budgets and boundaries.

newlib and the syscall boundary

The C standard library used here is newlib (its “nano” variant), a libc built for exactly this situation. Its portable core (printf formatting, malloc bookkeeping) is self-contained, but everything that would normally ask an OS for help funnels into a small set of stub functions — _write, _read, _sbrk, _open, … — that the platform must implement. Providing those stubs (and routing them somewhere useful, like a UART or an SD card) is most of what a bare-metal runtime is; soctglue's syscall handler chain below is its version of that boundary.

Harts start all at once

RISC-V calls a hardware thread a hart. On a multi-core SoC every hart comes out of reset simultaneously and races into the same entry point — there is no primary CPU that boots the others, unless software creates that convention. Startup code therefore elects a primary hart, parks the rest on a barrier, and releases them once the runtime is initialized.

Hardware discovery via the device tree

The SoC generator emits a different memory map per configuration, so binaries cannot hardcode addresses. Instead the boot flow passes a pointer to the device tree blob (DTB), the hardware self-description generated alongside the design, and the runtime parses it to find the UART, SD controller, memory size and CPU count at run time. One compiled program therefore runs unchanged on any generated variant of the SoC.

The device tree's journey: boot ROM, OpenSBI, Linux

For the Linux boot image (see Booting Linux below), the device tree is not consulted once — it flows down the whole boot chain, and each stage uses and reshapes the same tree. Nothing is ever literally merged from two trees; one blob is passed along and amended:

  1. Boot ROM. The generator's DTB is baked into the boot ROM, and sd-boot hands its address to whatever it loads in register a1 (hart id in a0) — the RISC-V boot convention every later stage speaks as well. Bare-metal programs consume this ROM copy directly.
  2. BOOT.ELF ignores the ROM copy. The boot image embeds its own DTB, compiled from the design's DTS at build time (OpenSBI's FW_FDT_PATH) with slack space added (dtc -p). Two reasons: the ROM copy is not writable, and the next stage must write to the tree; and the boot image becomes self-contained — it does not care which ROM DTB the flashed bitstream carries, only that the embedded one matches the flashed hardware.
  3. OpenSBI reads it… The firmware's own drivers are device-tree probed exactly like Linux's: the CLINT it uses for timers and IPIs and the UART behind its banner output come from the same nodes Linux will bind later — one description serving two consumers.
  4. …then relocates and fixes it up. OpenSBI copies the tree to a configured DRAM address (FW_PAYLOAD_FDT_ADDR) and edits it in place: it inserts reserved-memory nodes covering its own footprint — the memory it shields with PMP so the OS can never touch machine-mode firmware. These appear in a boot log as mmode_resv* regions. This is the closest thing to a “merge” in the chain: the hardware description gains firmware-owned annotations before the OS ever sees it.
  5. Linux receives the fixed-up tree — a1 again, at the S-mode kernel entry. The kernel parses the memory node into its zones (honoring the reservations from step 4), binds drivers by compatible, and reads device properties (the UART's current-speed, without which the console driver refuses to probe). The command line, too, comes from the tree: the generator emits a /chosen node whose bootargs name the design's UART as console and early console — hardware facts, kept with the hardware description. (A kernel can also carry a built-in CONFIG_CMDLINE; by default that is only a fallback for trees without /chosen, and this build leaves it unset so the tree is the single source.)

The corollary: the embedded DTB is a snapshot of one hardware generation. Rebuild the boot image whenever the design changes — a tree describing more memory than the flashed decode backs, or devices at moved addresses, fails at run time, not loudly at build time.

Directory layout

Programs and boot ROMs are auto-discovered subdirectories. Each directory's CMakeLists includes one of the shared templates — and which template it includes decides what kind of binary it becomes:

binaries/                          # The bare-metal CMake project (vendored riscv toolchain)
  bootroms/<name>/                 # Runs from on-chip ROM at reset (first stage)
  bootroms/template.cmake          # Template for boot ROMs (self-contained, no soctglue)
  baremetal/<name>/                # Runs directly on the SoC, no OS (M-mode, soctglue)
  baremetal/soctglue-static.cmake  # Template: bare-metal programs (soctglue + newlib)
  baremetal/xsdb-flash.cmake       # Included by the bare-metal template: JTAG flash targets
  linux/                           # A SEPARATE CMake project: everything Linux
    userspace/<name>/              # Runs in Linux userspace (real add_executable targets)
    userspace/userspace.cmake      # Template: static riscv64-linux-musl ELF
    userspace/initram.cmake        # Template: userspace + a bootable <name>.BOOT.ELF
    linux-stable/, opensbi/        # Drop-in kernel and OpenSBI source checkouts
    drivers/, rootfs/              # Reserved: kernel modules; dynamic userspace (see READMEs)
shared/soctglue/                   # Runtime library: soct.ld, crt0, syscall layer, FatFs glue

binaries/linux is its own project because CMake allows one toolchain per build tree, and the bare-metal toolchain cannot target Linux — see Booting Linux below.

Browse binaries/baremetal for the bundled examples and hardware-validation tools.

soctglue

soctglue is a static C library (libsoctglue.a) that every program links against. It provides the bare-metal runtime that sits between the hardware and your main().

Startup (crt0.S → _soct_start_main)

The assembly entry point _start (placed in .text.init) runs on every hart. It:

  1. Zeros all registers except a0 (hart ID) and a1 (DTB pointer), which the SoC passes at reset.
  2. Initializes the FPU if the ISA extension is present.
  3. Sets up the global pointer, thread pointer, and per-hart stack (each hart gets its own stack slice, sized by __stack_shift from the linker script).
  4. On the primary hart (hart 0 by default), zeros BSS and calls global constructors.
  5. Secondary harts spin on a boot barrier until the primary releases them.
  6. Calls _soct_start_main(hartid, dtb_ptr) on the primary hart.

_soct_start_main

Before calling main(), this function:

  1. Parses the DTB blob using the embedded smoldtb parser to discover peripherals.
  2. If a UART is found in the DTB, registers a UART syscall handler so printf works over serial.
  3. If an SD controller is found and SOCT_NEEDS_FATFS is set, mounts the SD volume via FatFs.
  4. If the HTIF device is present (Verilator simulation), registers an HTIF syscall handler and fetches argc/argv from the host.
  5. Releases secondary harts, then calls main().

Secondary harts jump to __main() if your program defines it, or spin in wfi otherwise.

Memory layout, heap and stacks (soct.ld + sbrk)

Programs are linked at the DRAM base (SOCT_MEM_BASE_ADDR, typically 0x8000_0000). The layout enforces a fixed program budget:

  • The program image (text, data, BSS) plus a minimum early heap of 128 KiB must fit into SOCT_PROGRAM_MAX_SIZE (default 4 MiB). If it does not, linking fails with a readable assertion — increase the budget with target_link_options(<prog> PRIVATE "LINKER:--defsym=SOCT_PROGRAM_MAX_SIZE=<bytes>") or shrink the program.
  • The per-hart stacks sit at the fixed boundary base + SOCT_PROGRAM_MAX_SIZE. Per-hart stack size is the next power of two above 24 KiB + TLS size; contiguous slices are allocated for up to SOCT_MAX_HARTS harts.
  • Everything above the stacks belongs to the heap. soctglue's sbrk serves three regions in order: the early heap between _end and the stacks, then the extended heap from the end of the stacks to the end of RAM as reported by the device tree (soct_ram_end()). On a 16 GiB board the heap capacity is ~16 GiB minus the program budget.
Allocator caveat: newlib's malloc (both the nano and the full allocator) rejects single requests of ≥ 2 GiB, and nano-malloc additionally cannot split a free chunk when the remainder would be ≥ 2 GiB. For huge allocations use several < 2 GiB blocks (see binaries/baremetal/heap-test, which probes exactly these limits).

Static stack-usage checking

The bare-metal template compiles everything with -fstack-usage and -Wstack-usage=<bytes> (default threshold 2048; override with -DSOCT_STACK_WARN=<bytes>). Any single function with a larger frame produces a compile-time warning, and the <prog>-info target (built by default) prints the largest collected frames alongside the .objdump/.nm files it writes. Runtime recursion is of course not caught.

Syscall handler chain

Standard C library calls (fopen, printf, read, …) are redirected through a handler table. The linker wraps puts, printf, vprintf, sprintf, snprintf, fopen, fopen64, freopen, and freopen64 so every I/O call passes through soctglue's dispatch instead of the newlib stubs.

The dispatch chain has two priority levels:

  • User handlers — registered by program code with soct_register_handler(). Tried first.
  • Default handlers — registered by soctglue internals (UART, HTIF) with soct_register_default_handler(). Tried only if every user handler passes.

Each handler inspects the syscall number and arguments and either handles the call (returning SOCT_HANDLER_HANDLED) or passes it on (SOCT_HANDLER_PASS). If no handler claims the syscall, errno is set to ENOSYS.

// Registering a custom handler from main():
static void handle_exit(soct_handler_resp_t *resp, sc_type_t syscall, ...) {
    if (syscall == SOCT_EXIT)
        printf("Exiting with code %llu\n", a0);
    resp->status = SOCT_HANDLER_PASS; // let default handler actually exit
}

soct_register_handler((soct_handler_t){ .handle = handle_exit });

FatFs integration (soct_ff)

When SOCT_NEEDS_FATFS is defined, soctglue compiles in soct_ff — a glue layer that bridges FatFs with the standard fopen/fread/fwrite/fclose API. Programs access the SD card using ordinary POSIX file calls:

FILE *f = fopen(SOCT_SD_PATH "/data.bin", "r");
fread(buf, 1, sizeof(buf), f);
fclose(f);

Without SOCT_NEEDS_FATFS, a stub is compiled instead that returns ENOSYS for all file operations.

The system file (SOCT_SYSTEM)

Every build requires a system file — a CMake file generated by SoCeteer that describes the target SoC. It provides variables such as:

The top-level CMakeLists.txt finds this file via FindSOCTSystem.cmake. It looks first at the SOCT_SYSTEM variable (passed with -DSOCT_SYSTEM=...), then falls back to a SOCTSystem-latest.cmake symlink in the repository root that SoCeteer creates automatically.

Adding a program

Create a subdirectory under binaries/baremetal/ with a CMakeLists.txt and your sources — the top-level build discovers it automatically, no registration needed. Then pick the template matching where the binary runs:

TemplateProjectProduces
soctglue-static.cmakebinaries/ A bare-metal ELF linked against soctglue and newlib, laid out by soct.ld, flashable over JTAG
linux/userspace/userspace.cmakebinaries/linux/ A static riscv64-linux-musl ELF (a real add_executable)
linux/userspace/initram.cmakebinaries/linux/ The same, plus a bootable <name>.BOOT.ELF with the program as the initramfs /init

Bare-metal programs (soctglue-static.cmake)

cmake_minimum_required(VERSION 3.20)

list(APPEND CMAKE_C_SRCS ${CMAKE_CURRENT_SOURCE_DIR}/main.c)
# list(APPEND CMAKE_CXX_SRCS ...) for C++ sources

include(${CMAKE_CURRENT_LIST_DIR}/../soctglue-static.cmake)

# Optional: add per-program flags after the include
target_compile_options(${SOCT_PROGRAM} PRIVATE -std=gnu11 -O2)

The entry point is the standard main(int argc, char **argv, char **envp). If the program uses multiple harts, also define __main(...) for secondary harts (same signature). The target is named after the directory, adjustable with SOCT_PROGRAM_PREFIX/SOCT_PROGRAM_SUFFIX (set before the include) — useful for building variants of the same program. The template ends by including xsdb-flash.cmake, so every bare-metal program has a flash target (see Flashing to an FPGA); the linked C library defaults to newlib-nano and can be swapped with SOCT_LIBC/SOCT_LIBCXX, the linker script with SOCT_LD_SCRIPT.

C++ programs

Bare-metal only: list sources in CMAKE_CXX_SRCS instead of (or in addition to) CMAKE_C_SRCS. The template detects the presence of C++ sources and links supc++/stdc++ automatically. Exceptions and RTTI are disabled. (The Linux userspace template is C-only.)

Conditionally building for a specific target

if (SOCT_TARGET STREQUAL "verilator")
    include(${CMAKE_CURRENT_LIST_DIR}/../soctglue-static.cmake)
    # ...
endif ()

Using the SD card / FatFs

Define SOCT_NEEDS_FATFS before the build (or in your program's CMakeLists.txt via target_compile_definitions) and access the SD volume with standard fopen/fread/fwrite. The volume root is SOCT_SD_PATH (defined to /SD by default with Linux-style volume IDs).

Linux userspace programs (binaries/linux)

Programs that run under the Linux kernel (see Booting Linux below) instead of on bare metal live in the separate binaries/linux project and follow the same directory-plus-template pattern with two templates:

cmake_minimum_required(VERSION 3.21)

list(APPEND CMAKE_C_SRCS ${CMAKE_CURRENT_SOURCE_DIR}/main.c)

include(${CMAKE_CURRENT_LIST_DIR}/../userspace.cmake)  # ELF only
# ...or ../initram.cmake to also get a bootable image (below)

The target (named after the directory) is a real add_executable — IDEs index it with full cross-compilation flags — producing a static riscv64-linux-musl ELF, copied to SOCT_ELFS_DIR. Static, because the target system is initramfs-only; there is no shared-library ecosystem to link against. soctglue, the bare-metal linker script and the syscall handler chain play no role here: these are ordinary Linux programs with a full C library. A program that includes initram.cmake instead additionally gets a <name>-boot-elf target: a complete <name>.BOOT.ELF with the program as the initramfs /init — running a userspace program on hardware becomes build, copy to SD, boot. The console-echo bring-up demo (linux/userspace/echo, which must never exit — it is PID 1) is the reference example. A boot image need not wrap a compiled program at all: linux/userspace/shell compiles nothing and instead hands its own initramfs description to soct_linux_add_boot_image(<name> DESC <desc> DEPENDS <packed files…>), packing the prebuilt BusyBox and a boot script (see Booting Linux). For IDE support, link binaries/linux as its own CMake project (CLion: Tools → CMake → Link CMake Project) configured with the same -DSOCT_SYSTEM.

Adding a boot ROM

Boot ROMs live under binaries/bootroms/ and follow the same auto-discovery pattern. Their CMakeLists.txt includes bootroms/template.cmake instead:

cmake_minimum_required(VERSION 3.20)

list(APPEND CMAKE_C_SRCS   ${CMAKE_CURRENT_SOURCE_DIR}/bootrom.c)
list(APPEND CMAKE_ASM_SRCS ${CMAKE_CURRENT_SOURCE_DIR}/head.S)

include(${CMAKE_CURRENT_LIST_DIR}/../template.cmake)

The bootrom template:

  • Links against the shared boot-ROM linker script (bootroms/bootrom.lds), which pins the layout every boot ROM shares: _start at the ROM base, the reset stub at the hart reset vector, and the embedded DTB at SOCT_DTB_ADDR — all three addresses come from the system file (derived from the same parameters that size the hardware ROM), and the link fails loudly if a stub outgrows its slot. Set SOCT_LD_SCRIPT before including the template to override.
  • Compiles the ELF with -Os -fno-pic.
  • Strips the ELF to a raw binary image with objcopy -O binary, rebased by the ROM base address (SOCT_BOOTROM_BASE_ADDR), placing the result at SOCT_BOOTROM_IMG.
  • Passes -DDEVICE_TREE=\"${SOCT_DTB}\" so the source can embed the DTB at compile time.

Boot ROMs do not link against soctglue — they are self-contained.

Building

cmake -S binaries -B build \
  -DSOCT_SYSTEM=/path/to/SOCTSystem.cmake \
  -DCMAKE_TOOLCHAIN_FILE=shared/cmake/toolchain-riscv.cmake
cmake --build build -j$(nproc)

ELF files land in SOCT_ELFS_DIR (set by the system file). Each bare-metal program's <name>-info target (built by default) writes an .objdump and .nm file alongside the ELF and prints the largest static stack frames.

Flashing to an FPGA

The bare-metal template ends by including xsdb-flash.cmake, so every bare-metal program gets a <name>-flash target that loads the compiled ELF onto the FPGA over JTAG using Xilinx xsdb — locally, or through an SSH host on which xsdb runs. The targets are only created when at least one of SOCT_FLASH_XSDB or SOCT_FLASH_HOST is set. The machinery is shared with the linux project (shared/cmake/SoctXsdbFlash.cmake), where the flashable artifact is the boot image: an initram.cmake program's <name>-flash target JTAG-loads its <name>.BOOT.ELF — a plain physical-address ELF — booting Linux with that program as /init without touching the SD card. A plain userspace ELF is still not flashable: JTAG writes physical memory with the hart halted and cannot deliver a file into a running kernel — such programs reach the target inside a boot image's initramfs.

cmake --build build --target hello-hart-flash

CMake variables

VariableDefaultDescription
SOCT_FLASH_XSDB(empty)Path to the xsdb binary (local, or on the remote host)
SOCT_FLASH_HOST(empty)SSH host to flash through — leave empty to flash locally
SOCT_FLASH_REMOTE_DIR/tmpDirectory on the remote host to upload the ELF into
SOCT_FLASH_BOOT_HART0Boot hart index written into register a0 before con

The DTB address written into register a1 is not a knob: it is the system file's SOCT_DTB_ADDR — the address the shared boot-ROM linker script pins the ROM's embedded device tree to, so the JTAG handoff matches what the boot ROM itself would pass.

Pass any of these at configure time:

cmake -S binaries -B build \
  -DSOCT_SYSTEM=/path/to/SOCTSystem.cmake \
  -DSOCT_FLASH_XSDB=/opt/Xilinx/Vivado/2024.2/bin/xsdb \
  -DSOCT_FLASH_BOOT_HART=0

Flash preludes (SOCT_FLASH_PRELUDE)

A program that needs extra steps before its ELF is loaded — board or subsystem initialization, staging auxiliary files — can set SOCT_FLASH_PRELUDE to a block of Python source before including the bare-metal template. It is injected into the generated flash wrapper and runs with elf, xsdb, host and remote_dir in scope, so one prelude serves local and remote flashing alike. The reference use is dp-test, whose prelude runs the Vivado-generated psu_init sequence the DisplayPort pipeline needs once per power-up.

Booting Linux (boot-elf)

The binaries/linux project — configured against the same SOCT_SYSTEM, with kernel and OpenSBI source trees dropped in as binaries/linux/linux-stable/ and binaries/linux/opensbi/ — produces bootable images: an OpenSBI fw_payload wrapping a Linux kernel (with an embedded initramfs) and the design's device tree, which the sd-boot boot ROM loads from the SD card like any other program. Its targets:

  • linux-sysroot — the riscv64-linux-musl sysroot (built automatically at configure time when missing, shared across build trees under shared/vendor/); see Linux userspace programs above.
  • boot-dtb — the design DTS compiled with fixup padding; it is embedded into the firmware, so boot images do not depend on the ROM copy of the DTB (but must match the flashed hardware).
  • <name>-boot-elf — for every program that includes initram.cmake: <name>.BOOT.ELF in SOCT_ELFS_DIR, a kernel with that program as /init. The kernel and OpenSBI build directories are shared: the ~2000 kernel objects compile once, and switching programs re-runs only the initramfs packing and the links (the initramfs description is a fixed path in the kernel build dir that each image overwrites — never a reconfigure). Image builds are serialized through a job pool; honored by the Ninja generator. With the SOCT_FLASH_* variables set, each image also gets a <name>-flash target that JTAG-loads it (see Flashing to an FPGA) — booting Linux without touching the SD card.
  • boot-elf — the canonical BOOT.ELF, under the name the boot ROM loads: a copy of the boot image selected by SOCT_BOOT_PROGRAM (default: shell).

There is no storage driver for the SD controller in mainline Linux, which is why the root filesystem is an initramfs embedded in the kernel image. The kernel configuration is the riscv defconfig plus linux/linux.fragment.in: the UART Lite console driver, the Xilinx interrupt-controller driver (all fabric interrupts cascade through an AXI INTC into the PLIC, whose gateways would lose the UART Lite's edge pulses — without this chained irqchip no peripheral interrupt, console included, ever fires), the Xilinx DMA engine driver for the video pipeline's VDMA, a pinned HZ, the initramfs, and AppArmor disabled (useless on this SoC, and its header generation is the one build step requiring GNU sed's \L on every host). The command line is deliberately not baked into the kernel: the generator emits a /chosen node whose bootargs select the design's UART as console and early console, so one kernel image serves any hardware generation.

The default image, shell.BOOT.ELF, boots into an interactive BusyBox shell. Its /init (linux/userspace/shell/init.sh) mounts the pseudo filesystems, installs the applet links and keeps a job-controlled shell alive on the console (setsid + cttyhack; the script itself stays PID 1, so an exiting shell is respawned instead of panicking the kernel). BusyBox is not built from source — that would add a second kbuild frontier for one binary — but taken from Debian's busybox-static riscv64 package: a static rv64gc binary, hash-pinned and fetched once at configure time into shared/vendor/busybox/ from snapshot.debian.org, whose archive URLs are permanent (regular pool URLs disappear once a package version is superseded).

Toolchains

Two toolchains split the work, deliberately:

  • LLVM/Clang builds the kernel and OpenSBI, through their upstream LLVM=<bindir>/ build mode. The vendored bare-metal binutils cannot link either one: their ld has no dynamic-linking support, OpenSBI links with -pie unconditionally, and a riscv MMU kernel must link its vDSO as a shared object (the failure shows up indirectly, as missing __vdso_*_offset symbols).
  • The repo's vendored toolchain builds everything bare-metal — programs and boot ROMs — and contributes one artifact to the Linux world: its rv64/lp64d libgcc.a, whose OS-agnostic builtins (soft fp128 arithmetic and friends) stand in for the compiler-rt that prebuilt LLVM does not ship for cross targets.

The LLVM toolchain is auto-detected by FindLLVMRiscv: candidates come from PATH, Homebrew (including its split llvm/lld keg-only formulas, for which a unified directory of symlinks is assembled), Debian's /usr/lib/llvm-* and MacPorts — and each candidate is proved by linking a RISC-V PIE through its ld.lld rather than trusted by name or version. That probe is why Apple's Xcode clang never qualifies: it ships without the RISC-V backend and without ld.lld. GNU make is found the same way (FindGNUMake: every gmake/gnumake/make candidate is version-checked, so an ancient system make cannot shadow a usable one). Userspace programs get their cross flags from the project's toolchain file (shared/cmake/toolchain-riscv-linux-musl.cmake), built on the same discovery. Any unmet requirement fails the binaries/linux configure loudly, with the remedy in the message; the bare-metal project is untouched by construction — the two worlds are separate build trees.

Host requirements

Every host:

  • LLVM with the RISC-V backend plus ld.lld and the llvm-* binutils (one directory or Homebrew's split installation).
  • GNU make ≥ 4.0 (the kernel refuses older), plus host flex, bison and bc; dtc for the device tree.
  • A build directory path without commas or spaces: OpenSBI passes linker scripts as -Wl,-T<path> and the compiler driver splits -Wl arguments at every comma, while GNU make cannot handle spaces in paths at all. Such paths are refused at configure time (IDE-generated build directories named after profiles are the usual offender).

Linux hosts: the distribution packages are sufficient — on Debian/Ubuntu: clang lld make flex bison bc device-tree-compiler. The repo's Docker image ships all of them; no accommodations of any kind are applied on Linux.

macOS (Darwin) hosts: four Homebrew packages — make, llvm, lld (a separate formula; llvm alone ships no linker) and gnu-sed. Beyond that, the build applies a set of accommodations automatically, all scoped to the kernel's host tools on Darwin only — target code and Linux hosts never see any of them:

AccommodationWhy
linux/host-compat/elf.h (verbatim musl, MIT) on the host include path macOS ships no <elf.h>; the kernel's host tools (scripts/elf-parse.h users, modpost) require one.
linux/host-compat/byteswap.h glibc-only header (bswap_*) used by modpost; shimmed with compiler builtins.
linux/host-compat/darwin-compat.h, force-included usr/gen_init_cpio.c uses O_LARGEFILE and copy_file_range, which Darwin lacks; defined to 0 and a read/write fallback respectively.
-D_UUID_T -D__GETHOSTUUID_H The macOS SDK typedefs uuid_t as a bare byte array (pulled in via unistd.h), colliding with the kernel's struct definition in scripts/mod/file2alias.c; the defines suppress the SDK's.
PATH shim mapping sed → gsed for kernel invocations BSD sed renders a replacement \t as a literal t, mangling every macro in the generated vDSO offsets header (__vdso_…_offsett0x…) into nonsense.
OpenSBI build directory pre-created BSD readlink -f fails on nonexistent paths, which derails OpenSBI's O= handling before the build starts.
