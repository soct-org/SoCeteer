The Vivado Block-Design DSL

SoCeteer describes Vivado block designs in Scala instead of hand-written TCL: components, pins, connections, clock domains, address assignment and constraints are ordinary objects, and a builder turns the resulting graph into the TCL/XDC collateral Vivado consumes. This guide covers the concepts you need to read the existing components and write your own.

Background

What a Vivado block design is and why it is scripted. Readers who use the IP integrator daily can skip ahead.

Block designs: wiring IP, not writing HDL

Alongside plain HDL sources, Vivado offers the IP integrator: a schematic-like block design in which pre-packaged, configurable hardware blocks (IP cores — DMA engines, interconnects, clock generators) are instantiated as cells and wired together. Cells expose two kinds of connection points: plain pins (single wires like a clock or reset) and interface pins — whole named bundles such as an AXI port, connected as one unit with protocol checking. Each IP is identified by its catalog VLNV (vendor:library:name:version) and configured through properties; Vivado derives some properties from context (e.g. clock frequencies propagate along connections), and derived properties are read-only — setting them is silently ignored or rejected, a recurring trap this DSL guards against.

The address map

A block design also carries the bus masters' view of memory: every slave's register window or memory range is assigned a base address and size in each master's address space (assign_bd_address). This map must agree with what the software expects — in SoCeteer's case, with RocketChip's diplomatic address map and the generated device tree.

Everything is TCL underneath

Every click in the Vivado GUI executes a TCL command, and the whole tool is drivable from TCL scripts alone. A design that exists only as GUI state is neither reviewable nor reproducible; a design generated as a deterministic TCL script can be diffed, versioned and recreated from scratch on any machine. That is the contract of this DSL: Scala builds a typed graph, and the graph is emitted as deterministic TCL — misconnections are caught by the Scala compiler or a generation-time check instead of by Vivado, hours later.

Constraints (XDC)

Synthesis needs facts that HDL cannot express: which package pin a port maps to, its electrical standard, and the timing relationships between clocks. These live in Xilinx Design Constraints (XDC) files, which address cells by instance name — the reason this DSL insists on stable, explicit instance names for anything a constraint might reference.

The builder and the graph

Everything registers itself with an implicit SOCTBdBuilder (injected via BdBuilderKey in the config). The builder is a directed multigraph: nodes are components, edges connect pins. Creating a component adds a node; the connect operators add edges. Nothing is emitted until the design is finalized and the generate* methods run — so construction order does not matter for the output (all emitted command groups are sorted).

Deliberate misuse fails loudly with a single exception type, VivadoDesignException — at generation time, not hours into synthesis.

Components

A component extends BdComp plus a kind marker (Markers.scala):

  • Xip — a Xilinx IP from the catalog; partName is the VLNV (e.g. xilinx.com:ip:smartconnect:1.0). The generated TCL verifies required IPs are installed before building.
  • IsModule — a module reference backed by your own HDL; the source files are shipped via dumpCollaterals (see the deinterleaver for the pattern: Verilog lives in the classpath resources and is copied next to the design sources at generation time).
  • XInlineHDL — Vivado's inline primitives (constants, slices, logic gates).

Pins are declared as objects inside the component; IP configuration comes from defaultProperties, which is evaluated at TCL-generation time and may inspect the graph (e.g. count connected pins). For the smallest complete example — a part name, three pin objects and two config properties — see RAMShiftReg; AXISmartConnect adds indexed pins and graph-dependent properties on top.

Instance names default to snake-cased class names with an index; use .withInstanceName("...") whenever a stable, readable name matters (timing constraints reference instance names!).

Components can be placed into a named block-design hierarchy with .withGroup("...") — purely organizational: the builder creates each distinct hierarchy once (create_bd_cell -type hier) before any member cell, member cells and their pin references become <group>/<cell>, and the netlist is unchanged. TCL that names cells directly must build paths from bdPath, never instanceName — this includes timing constraints, where paths containing a hierarchy separator must be matched as -hier -filter {NAME =~ *<path>} (a bare -hier search pattern silently matches nothing once the path contains /). Reference uses: each reset synchronizer plus its auto-generated fan-out slices forms its own block (periph_reset, core_reset, ddr_reset_<i>, pixel_reset), each memory channel bundles controller, SmartConnect, deinterleaver and reset gate as mem_<i>, and the PL video pipeline collapses into video.

Pins, ports and the connect operators

Endpoints carry a direction in their type: BdPinIn/BdPinOut/ BdPinInOut for scalar pins, BdIntfPin for interface pins (AXI, DDR4, clocks), and BdChiselPin for Chisel ports of the elaborated top (wrap them with portToBdPin(...)). Top-level interface ports are BdIntfPorts, parameterized by which side of the bus they are (PortMode) and how they come into existence (PortCreation): declared standalone with bus-default signal widths, or externalized from their connected pin when the widths depend on the IP configuration — the DDR4 custom interface uses the latter. Indexed pins (S00_AXI, aclk1, ...) come from pin factories: ACLK(0) creates the pin once and throws on re-creation; use getOrElseInit/get to share, and ACLK.next() to allocate the lowest free index when the position carries no meaning (the common case for interconnect ports) — reserve explicit indices for pins whose number is semantic, like concat inputs or interrupt slots.

Three operators (Connect.scala) add the edges, and the type system rejects impossible pairings at compile time with a detailed error message:

  • source --> sink — directional: a driver (DrivesNet) to a driven endpoint (DrivenByNet). Also accepts a collection on the right.
  • sink <-- source — the same, reversed.
  • a <-> b — directionless, for interface-to-interface connections (AXI master to slave, clock port to clock pin).

Custom pairings are added as implicit ToSinkConnect/AutoConnect instances in the component's companion — e.g. InlineConstant declares that its DOUT is the default source when you write TieOff() --> somePin.

Capabilities: constraints, addresses, finalization

  • EmitsConstraint — contribute XDC commands; constraints sharing an xdcName are grouped into one .xdc file (this is how the custom DDR4 interface ships its pin locations).
  • HasBdAddr — contribute assign_bd_address commands for the address map.
  • Finalizable — hook that runs once before TCL generation, for components that must inspect their connections to configure themselves (e.g. ProcSysReset inserts slices when a reset output fans out too wide).
  • MapsToPorts — register X_INTERFACE_INFO annotations that get patched into the top-level Verilog ports, so Vivado infers AXI/clock/reset interfaces from the raw wires.
  • bd.addTimingConstraints(...) / bd.addConfigTcl(...) — register lazily-evaluated TCL for the timing-constraints file and the end of the board script.

What gets generated

After finalizeDesign() locks the graph, the builder emits the board TCL in fixed phases — validate required IPs/modules, instantiate cells and ports, apply properties, connect nets, assign addresses, run registered config TCL, then validate_bd_design and wrapper generation — plus the init script, the timing constraints file and the grouped XDC files. If you want to see the output without a board run, the generated files land in workspace/<config>/<board>/vivado-srcs/.
