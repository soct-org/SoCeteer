Bare-Metal Programs & soctglue

Bare-metal programs and boot ROMs run directly on the SoCeteer-generated SoC, with no operating system underneath. They live in binaries/baremetal/ and binaries/bootroms/ and share a common runtime provided by soctglue. The build infrastructure both worlds share — the system file, building, JTAG flashing — is covered in the Binaries guide; programs that run under the Linux kernel are covered in Booting Linux.

Background

What “bare metal” actually implies. Readers who have written for microcontrollers can skip ahead.

No operating system means no environment

A bare-metal program is the only software on the machine. Nothing exists before its first instruction: no stack, no zeroed variables, no console, no files, no argv — the CPU simply starts executing at an address. Everything a hosted C program takes for granted must be established by the program itself, which is the job of the startup code (conventionally named crt0, “C runtime zero”): set up a stack, zero the BSS segment (the region holding zero-initialized globals, which only exists as a size in the binary), initialize globals, and only then call main().

The linker script decides where things live

With no loader and no virtual memory, the binary's layout is the memory layout: a linker script assigns each section (code, data, BSS) a physical address and defines symbols (like the end of the image) that the runtime uses to know where free memory begins. Stack and heap are not provided by anyone — they are just address ranges the linker script and runtime agree on, which is why this guide talks about budgets and boundaries.

newlib and the syscall boundary

The C standard library used here is newlib (its “nano” variant), a libc built for exactly this situation. Its portable core (printf formatting, malloc bookkeeping) is self-contained, but everything that would normally ask an OS for help funnels into a small set of stub functions — _write, _read, _sbrk, _open, … — that the platform must implement. Providing those stubs (and routing them somewhere useful, like a UART or an SD card) is most of what a bare-metal runtime is; soctglue's syscall handler chain below is its version of that boundary.

Harts start all at once

RISC-V calls a hardware thread a hart. On a multi-core SoC every hart comes out of reset simultaneously and races into the same entry point — there is no primary CPU that boots the others, unless software creates that convention. Startup code therefore elects a primary hart, parks the rest on a barrier, and releases them once the runtime is initialized.

Hardware discovery via the device tree

The SoC generator emits a different memory map per configuration, so binaries cannot hardcode addresses. Instead the boot flow passes a pointer to the device tree blob (DTB), the hardware self-description generated alongside the design, and the runtime parses it to find the UART, SD controller, memory size and CPU count at run time. One compiled program therefore runs unchanged on any generated variant of the SoC. The boot ROM bakes in the generator's DTB and hands its address to whatever it loads in register a1 (hart id in a0) — the RISC-V boot convention. Bare-metal programs consume this ROM copy directly; how the same tree travels on through OpenSBI into Linux is described in Booting Linux.

soctglue

soctglue is a static C library (libsoctglue.a) that every program links against. It provides the bare-metal runtime that sits between the hardware and your main().

Startup (crt0.S → _soct_start_main)

The assembly entry point _start (placed in .text.init) runs on every hart. It:

  1. Zeros all registers except a0 (hart ID) and a1 (DTB pointer), which the SoC passes at reset.
  2. Initializes the FPU if the ISA extension is present.
  3. Sets up the global pointer, thread pointer, and per-hart stack (each hart gets its own stack slice, sized by __stack_shift from the linker script).
  4. On the primary hart (hart 0 by default), zeros BSS and calls global constructors.
  5. Secondary harts spin on a boot barrier until the primary releases them.
  6. Calls _soct_start_main(hartid, dtb_ptr) on the primary hart.

_soct_start_main

Before calling main(), this function:

  1. Parses the DTB blob using the embedded smoldtb parser to discover peripherals.
  2. If a UART is found in the DTB, registers a UART syscall handler so printf works over serial.
  3. If an SD controller is found and SOCT_NEEDS_FATFS is set, mounts the SD volume via FatFs.
  4. If the HTIF device is present (Verilator simulation), registers an HTIF syscall handler and fetches argc/argv from the host.
  5. Releases secondary harts, then calls main().

Secondary harts jump to __main() if your program defines it, or spin in wfi otherwise.

Memory layout, heap and stacks (soct.ld + sbrk)

Programs are linked at the DRAM base (SOCT_MEM_BASE_ADDR, typically 0x8000_0000). The layout enforces a fixed program budget:

  • The program image (text, data, BSS) plus a minimum early heap of 128 KiB must fit into SOCT_PROGRAM_MAX_SIZE (default 4 MiB). If it does not, linking fails with a readable assertion — increase the budget with target_link_options(<prog> PRIVATE "LINKER:--defsym=SOCT_PROGRAM_MAX_SIZE=<bytes>") or shrink the program.
  • The per-hart stacks sit at the fixed boundary base + SOCT_PROGRAM_MAX_SIZE. Per-hart stack size is the next power of two above 24 KiB + TLS size; contiguous slices are allocated for up to SOCT_MAX_HARTS harts.
  • Everything above the stacks belongs to the heap. soctglue's sbrk serves three regions in order: the early heap between _end and the stacks, then the extended heap from the end of the stacks to the end of RAM as reported by the device tree (soct_ram_end()). On a 16 GiB board the heap capacity is ~16 GiB minus the program budget.
Allocator caveat: newlib's malloc (both the nano and the full allocator) rejects single requests of ≥ 2 GiB, and nano-malloc additionally cannot split a free chunk when the remainder would be ≥ 2 GiB. For huge allocations use several < 2 GiB blocks (see binaries/baremetal/heap-test, which probes exactly these limits).

Static stack-usage checking

The bare-metal template compiles everything with -fstack-usage and -Wstack-usage=<bytes> (default threshold 2048; override with -DSOCT_STACK_WARN=<bytes>). Any single function with a larger frame produces a compile-time warning, and the <prog>-info target (built by default) prints the largest collected frames alongside the .objdump/.nm files it writes. Runtime recursion is of course not caught.

Syscall handler chain

Standard C library calls (fopen, printf, read, …) are redirected through a handler table. The linker wraps puts, printf, vprintf, sprintf, snprintf, fopen, fopen64, freopen, and freopen64 so every I/O call passes through soctglue's dispatch instead of the newlib stubs.

The dispatch chain has two priority levels:

  • User handlers — registered by program code with soct_register_handler(). Tried first.
  • Default handlers — registered by soctglue internals (UART, HTIF) with soct_register_default_handler(). Tried only if every user handler passes.

Each handler inspects the syscall number and arguments and either handles the call (returning SOCT_HANDLER_HANDLED) or passes it on (SOCT_HANDLER_PASS). If no handler claims the syscall, errno is set to ENOSYS.

// Registering a custom handler from main():
static void handle_exit(soct_handler_resp_t *resp, sc_type_t syscall, ...) {
    if (syscall == SOCT_EXIT)
        printf("Exiting with code %llu\n", a0);
    resp->status = SOCT_HANDLER_PASS; // let default handler actually exit
}

soct_register_handler((soct_handler_t){ .handle = handle_exit });

FatFs integration (soct_ff)

When SOCT_NEEDS_FATFS is defined, soctglue compiles in soct_ff — a glue layer that bridges FatFs with the standard fopen/fread/fwrite/fclose API. Programs access the SD card using ordinary POSIX file calls:

FILE *f = fopen(SOCT_SD_PATH "/data.bin", "r");
fread(buf, 1, sizeof(buf), f);
fclose(f);

Without SOCT_NEEDS_FATFS, a stub is compiled instead that returns ENOSYS for all file operations.

You can override the linker script by setting SOCT_LD_SCRIPT before including the bare-metal template.

Adding a program

Create a subdirectory under binaries/baremetal/ with a CMakeLists.txt and your sources — the top-level build discovers it automatically, no registration needed. The CMakeLists includes the bare-metal template (for the other templates and worlds, see the Binaries guide):

cmake_minimum_required(VERSION 3.20)

list(APPEND CMAKE_C_SRCS ${CMAKE_CURRENT_SOURCE_DIR}/main.c)
# list(APPEND CMAKE_CXX_SRCS ...) for C++ sources

include(${CMAKE_CURRENT_LIST_DIR}/../soctglue-static.cmake)

# Optional: add per-program flags after the include
target_compile_options(${SOCT_PROGRAM} PRIVATE -std=gnu11 -O2)

The entry point is the standard main(int argc, char **argv, char **envp). If the program uses multiple harts, also define __main(...) for secondary harts (same signature). The target is named after the directory, adjustable with SOCT_PROGRAM_PREFIX/SOCT_PROGRAM_SUFFIX (set before the include) — useful for building variants of the same program. The template ends by including xsdb-flash.cmake, so every bare-metal program has a flash target (see Flashing to an FPGA); the linked C library defaults to newlib-nano and can be swapped with SOCT_LIBC/SOCT_LIBCXX, the linker script with SOCT_LD_SCRIPT.

Browse binaries/baremetal for the bundled examples and hardware-validation tools.

C++ programs

List sources in CMAKE_CXX_SRCS instead of (or in addition to) CMAKE_C_SRCS. The template detects the presence of C++ sources and links supc++/stdc++ automatically. Exceptions and RTTI are disabled. (The Linux userspace template is C-only.)

Conditionally building for a specific target

if (SOCT_TARGET STREQUAL "verilator")
    include(${CMAKE_CURRENT_LIST_DIR}/../soctglue-static.cmake)
    # ...
endif ()

Using the SD card / FatFs

Define SOCT_NEEDS_FATFS before the build (or in your program's CMakeLists.txt via target_compile_definitions) and access the SD volume with standard fopen/fread/fwrite. The volume root is SOCT_SD_PATH (defined to /SD by default with Linux-style volume IDs).

Adding a boot ROM

Boot ROMs live under binaries/bootroms/ and follow the same auto-discovery pattern. Their CMakeLists.txt includes bootroms/template.cmake instead:

cmake_minimum_required(VERSION 3.20)

list(APPEND CMAKE_C_SRCS   ${CMAKE_CURRENT_SOURCE_DIR}/bootrom.c)
list(APPEND CMAKE_ASM_SRCS ${CMAKE_CURRENT_SOURCE_DIR}/head.S)

include(${CMAKE_CURRENT_LIST_DIR}/../template.cmake)

The bootrom template:

  • Links against the shared boot-ROM linker script (bootroms/bootrom.lds), which pins the layout every boot ROM shares: _start at the ROM base, the reset stub at the hart reset vector, and the embedded DTB at SOCT_DTB_ADDR — all three addresses come from the system file (derived from the same parameters that size the hardware ROM), and the link fails loudly if a stub outgrows its slot. Set SOCT_LD_SCRIPT before including the template to override.
  • Compiles the ELF with -Os -fno-pic.
  • Strips the ELF to a raw binary image with objcopy -O binary, rebased by the ROM base address (SOCT_BOOTROM_BASE_ADDR), placing the result at SOCT_BOOTROM_IMG.
  • Passes -DDEVICE_TREE=\"${SOCT_DTB}\" so the source can embed the DTB at compile time.

Boot ROMs do not link against soctglue — they are self-contained.
