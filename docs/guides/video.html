DisplayPort Video

How a framebuffer in DRAM becomes pixels on a monitor: the hardware pipeline the video config adds to the design, the contract software has to fulfill to light it up, and the constraints that fall out of both.

Background

The concepts this guide builds on. Readers familiar with raster video can skip ahead.

Framebuffers and scanout

A framebuffer is a region of ordinary RAM holding one value per screen pixel — here three bytes (red, green, blue), so a 1280×720 frame occupies about 2.7 MB. A monitor is not sent a frame once when it changes: the display is redrawn continuously, top to bottom, at the refresh rate. Display hardware therefore scans out the framebuffer forever — reading pixels row by row, left to right, 60 times per second for a 60 Hz mode — whether or not anything changed. Scanout is a hard real-time consumer: a pixel that arrives late is not displayed late, it is displayed wrong (or the monitor drops the signal entirely). That is why this pipeline is designed around sustained bandwidth rather than average throughput.

Video timing: the active area is not the whole frame

A video mode transmits more than its visible pixels. Each line consists of the active pixels plus a horizontal blanking interval, and each frame consists of the active lines plus vertical blanking. Blanking is a CRT inheritance (time for the electron beam to fly back), but every digital standard keeps it — sinks use it to synchronize, and software uses vertical blanking as the safe moment to switch buffers. Blanking is subdivided into the front porch (after the last active pixel), the sync pulse and the back porch (before the next active pixel). Three signals accompany the pixel data: hsync and vsync pulse once per line/frame, and data enable (DE) is high exactly during active pixels.

The sum of active and blanking pixels gives the total frame size. For 1280×720 at 60 Hz (a CTA-861 mode) each line is 1280 active + 370 blanking = 1650 pixels, and each frame is 720 active + 30 blanking = 750 lines. These totals define the pixel clock: 1650 × 750 × 60 ≈ 74.25 MHz — one pixel must be produced on every single cycle of that clock. A mode is therefore not just a resolution; it fixes an exact clock frequency and an exact blanking structure that transmitter and sink must agree on.

Double buffering

Drawing into the framebuffer that is currently being scanned out produces tearing: the top of the screen shows the old frame, the bottom the new one, with a visible seam that moves. The standard fix is double buffering — draw into a second buffer while the first is displayed, then switch the scanout pointer during vertical blanking. The switch must take effect only at a frame boundary; hardware that latches a new address mid-frame tears just the same.

Why a DMA engine

At 1280×720×60 the display consumes ~166 MB/s — every second, forever. No CPU should spend its cycles copying that, and a CPU also cannot guarantee the hard per-cycle delivery scanout demands. A dedicated DMA engine (here the Xilinx VDMA) re-reads the framebuffer from memory every frame and emits it as a pixel stream, with an internal FIFO to ride out memory-latency spikes. The CPU only draws pixels into memory and flips buffers.

DisplayPort is a packet link, not a pixel bus

Older display interfaces (VGA, DVI, parallel RGB) transmit the pixel clock and sync signals more or less directly. DisplayPort does not: it is a packetized, fixed-rate serial link (similar in spirit to PCIe) whose link rate is unrelated to the pixel rate. Before any video flows, the transmitter must train the link (negotiate lane count, speed and equalization with the monitor) and describe the video inside it via the Main Stream Attributes (MSA) — resolution, blanking, and an M/N clock ratio from which the sink regenerates the pixel clock. This is why a DisplayPort output needs a driver where a VGA output needs a wire, and why the software contract below has a PS-side half at all.

Enabling the pipeline

sbt "runMain soct.SOCTLauncher --target vivado --board ZCU104 \
  --with-config soct.WithVideoStream"

The config adds the video mode's parameters (VideoStreamParams, default 1280×720@60) and one fabric interrupt for the VDMA (cascaded, like every peripheral interrupt, through the AXI interrupt controller into the PLIC; the device tree carries it on the VDMA's channel child node, following the mainline binding). The ZCU104 is the reference board throughout this guide; --board takes any registered MPSoC board (see Adding an FPGA Board).

Generation validates that the mode is drivable: the frame fetch must sustain width × height × fps × 3 bytes per second through the coherent DMA path. That path moves 8 bytes per periphery-clock cycle in theory and roughly a quarter of that in practice (measured on the reference ZCU104 at the default 100 MHz: ~187 MB/s, where 1080p60 needs 373 MB/s and delivered 30 of 60 frames per second). The check scales with the configured periphery clock (--periphery-freq-mhz) — a faster clock admits larger modes. An undrivable mode fails the build with the numbers instead of failing on the monitor. Independently, each mode needs a known pixel clock: the supported modes are enumerated in SOCTVivadoSystemBase.pixelClockFor (1280×720@60 and 1920×1080@60); a new mode is added by giving it its CTA-861 pixel clock there.

The hardware pipeline

The pipeline targets Zynq UltraScale+ MPSoC boards. On those, the DisplayPort connector is driven by the PS, not by PL pins: the DP controller is hardened in the PS and reaches the connector over the PS-GTR serial lanes. The PL therefore streams pixels into that controller's live video input, and the RISC-V programs it through an address window. Everything is instantiated by wireVideoStream:

  • AXIVideoDMA reads frames from DRAM and emits them as an AXI4-Stream of 24-bit RGB pixels. With WithVideoStream it fetches over the same coherent DMA path the SD card uses (through the Rocket L2 frontend, so CPU writes to the framebuffer need no cache maintenance); with WithIncoherentVideoStream it masters the memory controller directly — see Coherent and incoherent frame fetch below.
  • AxisVideoOut converts the stream to parallel video (data + syncs + data-enable), aligned to the timing of the VideoTimingController. It runs in master timing mode: it gates the timing generator until its FIFO is primed and releases it at a frame boundary, which is what guarantees output lock.
  • The parallel video feeds the ZynqUltraPS live video pins — detailed port by port below.
  • An AxiAddrOffset window makes the PS register space reachable from the MMIO port: a 16 MiB window maps onto the PS-fixed addresses of the DP controller, its SERDES and the AV buffer manager. Rocket's MMIO decode cannot cover those addresses directly without colliding with the DRAM decode.
  • An AxiGpio exposes the video out's locked/underflow/overflow pins as a readable register — the pipeline's health, since those signals otherwise never leave the fabric.

The MPSoC as a peripheral

The design uses the zynq_ultra_ps_e IP purely as a peripheral of the PL: its APU cores never run. At generation time the board preset is applied (clocking, MIO, the DP lane and refclk setup on the PS-GTR), then the design enables exactly the two faces it needs — PSU__USE__VIDEO and PSU__USE__S_AXI_GP6 (32-bit data width) — and disables the preset's unused PL interfaces (M_AXI_HPM0/1_FPD, S_AXI_HP0_FPD) so nothing is left unclocked. This PSU configuration is fixed in the bitstream; what software can and cannot change is listed per port:

S_AXI_LPD + saxi_lpd_aclk — the register window

The PS's low-power-domain AXI slave (hardware name SAXIGP6). This is how the RISC-V reaches every PS register: Rocket MMIO port → MMIO SmartConnect → AxiAddrOffset (adds 0x8000_0000) → S_AXI_LPD. A load from 0x7D4A_0000 therefore arrives at the DP controller at 0xFD4A_0000; the LPD's internal routing carries it onward into the FPD, where the DP registers physically live. The window base, size and offset are fixed hardware; everything behind the window is software: the DP controller (link training, main stream attributes), its SERDES lane registers and the AV buffer manager are all programmed by the driver through this one port. The port is clocked by the periphery clock on the PL side; its PS-side width registers (AFIFM) are written by psu_init — before that runs, the very first access stalls the bus.

dp_video_in_clk — the pixel clock into the PS

Carries the PL-generated pixel clock (the MMCM output, 74.25 MHz for 720p60) into the DP subsystem. The frequency is fixed at generation time by the video mode; software cannot change it, but software must select it: the AV buffer manager has a video-clock mux (PS internal clock vs. PL clock), and the driver switches it to the PL side for live video. Until that selection, the PS video pipeline runs on the wrong clock and ignores the live input.

dp_live_video_in_pixel1[35:0] — pixel data

The live pixel bus, 12 bits per component. The video out emits 24-bit RGB (8 bits per component, UG934 ordering); three slices and a concat pad each component with four zero LSBs, so the data sits MSB-aligned in the PS's 12-bit lanes. This mapping is fixed wiring — but its interpretation is software: the AV buffer manager's live-input format register must be set to RGB, 8 bits per component, or the PS scales and swizzles the values incorrectly.

dp_live_video_in_hsync / vsync / de — timing

The video out's sync outputs and its vid_active_video (as data enable) drive these directly. The PS samples all of them on dp_video_in_clk. Polarities follow the video mode (both positive for the CEA modes used here) and are produced by the timing controller, which software programs — so the timing itself is software-defined, but it must match the mode advertised in the device tree, because the pixel clock is sized for exactly that mode. The main stream attributes the driver writes into the DP controller must describe the same timing; the DP transmitter frames whatever the live interface delivers.

What remains PS-internal

Between the live input and the DP connector sit the AV buffer manager (input select, blender, background color) and the DP transmitter (link training, MSA, the PS-GTR serdes) - all software-programmed through the window, none of it wired to the PL. The DP link clock derives from the PS-GTR reference clock, which the board preset routes and psu_init configures; it is unrelated to the pixel clock, which is why the DP protocol carries M/N clock-ratio values for the sink.

Clocking

Three clock domains meet in this pipeline:

  • Periphery clock (configurable with --periphery-freq-mhz, default 100 MHz; synthesized by the system clock wizard, so the requested frequency is not limited to what the memory controller's clocking offers): used by the VDMA's AXI4-Lite and memory-read interfaces, the timing controller's and status GPIO's AXI ports, the address-offset window and saxi_lpd_aclk. It also feeds the pixel MMCM's input.
  • Pixel clock (MMCM output; 74.25 MHz for 720p60): the whole video path — the VDMA's stream side, the video out (both its stream and video-IO sides), the timing controller's timing side and dp_video_in_clk. The stream carries one pixel per cycle, so this clock is the pixel rate; running the stream side on the slower periphery clock starves the video out mid-line (measured, not theoretical). The VDMA bridges the two domains internally (its independent-clocks mode is derived by Vivado from these connections).
  • PS-internal clocks: the DP video pipeline runs on the live pixel clock once software selects the PL source; the DP link and its SERDES run from the PS-GTR reference configured by psu_init.

Reset behavior

Two reset synchronizers govern the PL side; the conditions under which each part of the subsystem is held in reset:

  • Periphery domain (board reset, or the debug module's ndreset): holds the VDMA (axi_resetn), the timing controller's and status GPIO's AXI interfaces, and — via its active-high output — the pixel MMCM itself. A periphery reset therefore also collapses the pixel clock.
  • Pixel domain (its own synchronizer): held while the periphery is in reset or the MMCM is unlocked (dcm_locked). It resets only the video out; release is synchronized to the pixel clock. Its external reset input is fed the active-low periphery reset deliberately: the signal passes through polarity-stripping fabric, and Vivado then infers the (read-only) pin polarity as active-low — feeding the active-high reset holds the domain in permanent reset. See the warning on ProcSysReset.
  • Never reset: the timing controller's pixel side and the video out's video-IO side have their resets tied inactive and their enables tied high. They free-run from the moment the pixel clock exists and merely produce garbage until the driver programs the VDMA and the timing — which is harmless, because the video out refuses to assert locked (and the DP shows the blender background) until the stream and timing agree.
  • PS side: PL resets never touch it. The PS comes out of reset through psu_init once per power-up, and the DP controller additionally has a register-level soft reset that the driver pulses when it reprograms the stream.

Coherent and incoherent frame fetch

The frame fetch can reach DRAM two ways, and the choice is a real trade, not a detail.

soct.WithVideoStream routes the VDMA through the Rocket L2 frontend, the same coherent port the SD card uses. Frames the CPU writes are visible to the DMA with no cache maintenance at all. The cost is that the coherent port fragments the DMA's bursts and caps its in-flight transactions, so throughput becomes latency-bound: at 720p60 the fetch needs 166 MB/s and idles at almost exactly that, leaving no margin. Any CPU traffic queues ahead of it and pushes the round trip over the edge — measured on a ZCU104, a streaming CPU load of ~50 MB/s was enough to halve delivered frames and unlock the video output essentially all the time. This is a starvation problem, not a bandwidth one: the numbers involved are a small fraction of what the DDR4 can move.

soct.WithIncoherentVideoStream gives the VDMA its own port on the memory controller's SmartConnect instead, bypassing the coherent path entirely. Under the same CPU ramp the fetch holds its full frame rate with the video output locked throughout. The price is that DRAM is no longer coherent with the CPU's caches for framebuffer contents: rendered pixels sit dirty in the write-back L1 (and L2, if present) and the DMA would fetch stale memory. Software must push them out, so the design tree marks such a pipeline soct,incoherent on the VDMA node.

Because it removes the coupling rather than merely weakening it, the incoherent variant is also what makes an L2 affordable in a video design: soct.WithL2Cache can be added for CPU performance without the frame fetch paying for it. Both variants suffix the workspace directory (-video / -video-nc, plus -l2).

Two mechanisms cover the flush, picked from what the design has — there is no Zicbom on this Rocket, so neither is a cache-block instruction:

  • With an L2, write each line's physical address to the InclusiveCache Flush64 register. The write blocks until the line is out, and because the cache is inclusive it also pulls the line from L1.
  • Without one, read a buffer the size of the L1 (16 KiB): every set gets refilled, so the framebuffer lines are evicted, and with no L2 behind them they land in DRAM.

The Flush64 cost is proportional to the region, and that matters more than it looks: a 720p frame is 43,200 cache lines, while an animation usually redraws a small part of one. Flush the dirty rectangle, not the frame — for dp-test's teapot that is a 288×288 box, about 4,000 lines, an order of magnitude less work per frame. The whole-cache fallback is address-independent, so it is a single sweep either way.

The software contract

dp-test is the reference implementation of everything below, in the order listed. It doubles as the pipeline's diagnostic tool and ends in a rotating-teapot demo. Its main.c holds the program's flow — discovery, bring-up, then the demo loop — and the stages it drives sit beside it: video.c for the whole PL side (framebuffers, VDMA, timing generator), dp.c for the PS side, render.c for the test pattern and the teapot, and diag.c for the instrumentation.

0. Initialize the PS (once per power-up)

The DP controller's clocks and the PS-GTR SERDES are configured by the psu_init.tcl that Vivado generates with the design. Without it, the very first windowed register access stalls the bus. In xsdb:

targets -set -filter {name =~ {PSU*}}
source .../ip/<design>_zynq_ultra_ps_0_0/psu_init.tcl
psu_init
psu_post_config
psu_ps_pl_isolation_removal
psu_ps_pl_reset_config

The dp-test flash target runs this automatically before every flash (see its CMakeLists).

1. Discover the pipeline from the device tree

The design describes itself; nothing is hardcoded in software:

  • vdma0 / vtc0 / vidstat0 — register bases of the VDMA, the timing controller and the status GPIO.
  • vtc0 also carries soct,hactive, soct,vactive and soct,fps: the video mode baked into the design's pixel clock. Software must program exactly this timing.
  • dpwin0 (compatible soct,zynqmp-dp-window) — the window's base and size, plus soct,ps-base, the PS address its start maps to. Software reaches any documented PS register at window base + (PS address − ps-base).

2. Program the PL: timing and frames

The timing controller is programmed with the mode's blanking intervals (the register sequence follows the Xilinx vtc driver's generator setup) and the VDMA runs in register-direct park mode: its frame stores point at CPU-drawn framebuffers, and retargeting the park pointer to another store flips the displayed buffer at the next frame boundary — that is the tear-free page flip. Three caveats all live in video.c's comments:

  • Frame stores are latched when VSIZE is written; rewriting VSIZE on a running channel visibly glitches the frame being scanned out. Point the stores at their buffers once and flip only the park pointer.
  • DMACR's Circular_Park bit resets to 1, i.e. to circular mode, in which the park pointer is ignored and every frame store is scanned out in turn. It must be cleared explicitly; leaving it set shows the hidden buffer every third frame.
  • The VDMA read master is 32-bit: framebuffers must live below 4 GiB, i.e. in the first 2 GiB of DRAM.

3. Program the PS: routing and the DP link

The PS side is driven by the vendored Xilinx drivers in dp-test/xilinx (dppsu for the DP controller, avbuf for the AV buffer manager), kept byte-for-byte unmodified. The shim layer in dp-test/soct_xil provides the BSP headers they include by name — crucially Xil_In32/Out32, which translate the drivers' hardcoded PS addresses through the window — so the drivers never know they are running on a RISC-V. The sequence: select the live input and the PL video clock in the AV buffer manager, initialize the DP TX, wait for a monitor (HPD), train the link at the maximum common capabilities, write the main stream attributes for the advertised mode, and enable the main stream.

4. Verification

The status GPIO tells the truth about the PL side: locked means video is leaving the fabric; underflow means the frame fetch is starving (bandwidth). On the PS side, the DP controller's interrupt status reports vertical-blanking events of the transmitted stream. dp-test checks all of these and prints a verdict for each stage, so a failure is localized to one link of the pipeline instead of presenting as a black screen.
