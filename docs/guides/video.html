DisplayPort Video

How a framebuffer in DRAM becomes pixels on a monitor: the hardware pipeline the video config adds to the design, the contract software has to fulfill to light it up, and the constraints that fall out of both.

Enabling the pipeline

sbt "runMain soct.SOCTLauncher --target vivado --board ZCU104 \
  --with-config soct.WithVideoStream"

The config adds the video mode's parameters (VideoStreamParams, default 1280×720@60) and one PLIC interrupt for the VDMA. Designs without the config are byte-identical to before — the whole pipeline is gated on HasVideoStream.

Generation validates that the mode is drivable: the frame fetch must sustain width × height × fps × 3 bytes per second through the coherent DMA path, which delivers roughly a quarter of its theoretical 8 bytes/cycle in practice (measured on the ZCU104 at 100 MHz: ~187 MB/s, where 1080p60 would need 373 MB/s and got 30 of 60 frames per second). An undrivable mode fails the build with the numbers instead of failing on the monitor.

The hardware pipeline

The ZCU104 routes its DisplayPort connector to the PS (the MPSoC's DP controller on PS-GTR lanes), not to PL pins. The PL therefore streams pixels into the PS DP controller's live video input, and the RISC-V programs that controller through an address window. Everything below is instantiated by wireVideoStream:

  • AXIVideoDMA reads frames from DRAM over the same coherent DMA path the SD card uses (through the Rocket L2 frontend, so CPU writes to the framebuffer need no cache maintenance) and emits them as an AXI4-Stream of 24-bit RGB pixels.
  • AxisVideoOut converts the stream to parallel video (data + syncs + data-enable), aligned to the timing of the VideoTimingController. It runs in master timing mode: it gates the timing generator until its FIFO is primed and releases it at a frame boundary, which is what guarantees output lock.
  • The parallel video feeds the ZynqUltraPS live video pins, 8 bits per component padded to the PS's 12-bit lanes.
  • An AxiAddrOffset window makes the PS register space reachable from the MMIO port: a 16 MiB window maps onto the PS-fixed addresses of the DP controller, its SERDES and the AV buffer manager. Rocket's MMIO decode cannot cover those addresses directly without colliding with the DRAM decode.
  • An AxiGpio exposes the video out's locked/underflow/overflow pins as a readable register — the pipeline's health, since those signals otherwise never leave the fabric.

Clocks and resets

Control and memory sides run on the periphery clock. The stream side — the VDMA's pixel stream, the video out, the timing generator and the PS live input — runs on a dedicated pixel clock synthesized by an MMCM (74.25 MHz for 720p60): the stream carries one pixel per cycle, so its clock must be the pixel rate. The pixel domain has its own reset synchronizer held until the MMCM locks; its external reset is fed active-low deliberately — see the warning on ProcSysReset about Vivado's polarity inference through slices, which cost this pipeline a bring-up cycle.

The software contract

dp-test is the reference implementation of everything below, in the order listed. It doubles as the pipeline's diagnostic tool and ends in a rotating-teapot demo.

0. Initialize the PS (once per power-up)

The DP controller's clocks and the PS-GTR SERDES are configured by the psu_init.tcl that Vivado generates with the design. Without it, the very first windowed register access stalls the bus. In xsdb:

targets -set -filter {name =~ {PSU*}}
source .../ip/<design>_zynq_ultra_ps_0_0/psu_init.tcl
psu_init
psu_post_config
psu_ps_pl_isolation_removal
psu_ps_pl_reset_config

The dp-test flash target runs this automatically before every flash (see its CMakeLists).

1. Discover the pipeline from the device tree

The design describes itself; nothing is hardcoded in software:

  • vdma0 / vtc0 / vidstat0 — register bases of the VDMA, the timing controller and the status GPIO.
  • vtc0 also carries soct,hactive, soct,vactive and soct,fps: the video mode baked into the design's pixel clock. Software must program exactly this timing.
  • dpwin0 (compatible soct,zynqmp-dp-window) — the window's base and size, plus soct,ps-base, the PS address its start maps to. Software reaches any documented PS register at window base + (PS address − ps-base).

2. Program the PL: timing and frames

The timing controller is programmed with the mode's blanking intervals (the register sequence follows the Xilinx vtc driver's generator setup) and the VDMA runs in register-direct park mode: its frame stores point at CPU-drawn framebuffers, and retargeting the park pointer to another store flips the displayed buffer at the next frame boundary — that is the tear-free page flip. Two caveats both live in dp-test's comments:

  • Frame stores are latched when VSIZE is written; rewriting VSIZE on a running channel visibly glitches the frame being scanned out. Point the stores at their buffers once and flip only the park pointer.
  • The VDMA read master is 32-bit: framebuffers must live below 4 GiB, i.e. in the first 2 GiB of DRAM.

3. Program the PS: routing and the DP link

The PS side is driven by the vendored Xilinx drivers in dp-test/xilinx (dppsu for the DP controller, avbuf for the AV buffer manager), kept byte-for-byte unmodified. The shim layer in dp-test/soct_xil provides the BSP headers they include by name — crucially Xil_In32/Out32, which translate the drivers' hardcoded PS addresses through the window — so the drivers never know they are running on a RISC-V. The sequence: select the live input and the PL video clock in the AV buffer manager, initialize the DP TX, wait for a monitor (HPD), train the link at the maximum common capabilities, write the main stream attributes for the advertised mode, and enable the main stream.

4. Verify, don't hope

The status GPIO tells the truth about the PL side: locked means video is leaving the fabric; underflow means the frame fetch is starving (bandwidth). On the PS side, the DP controller's interrupt status reports vertical-blanking events of the transmitted stream. dp-test checks all of these and prints a verdict for each stage, which is how every failure mode of this pipeline was localized during bring-up.
