Architecture & Hardware Flow

What happens between sbt "runMain soct.SOCTLauncher ..." and a pile of Verilog, TCL and a boot ROM image: the launcher pipeline, how configs shape the design, how the transpiler lowers Chisel to Verilog, and how the boot ROM gets built during elaboration.

Background

The concepts this pipeline is built from. Readers who know the Chisel ecosystem can skip ahead.

Hardware generators, not hardware descriptions

Chisel is not a language that is compiled to a chip; it is a Scala library for constructing circuits. Running a Chisel program (called elaboration) executes ordinary Scala code whose side effect is building a graph of hardware nodes — registers, wires, module instances. Because the constructing code is a real program, it can loop, branch and take parameters, which makes a Chisel design a generator: one codebase that emits a different circuit for every parameter set (number of cores, cache sizes, which peripherals exist). Everything in this guide happens at elaboration time, on the development machine — no hardware is involved yet.

FIRRTL and firtool

Elaboration serializes the circuit graph as FIRRTL, an intermediate representation for register-transfer-level hardware — the same role an IR plays in a software compiler. A separate compiler then lowers FIRRTL to synthesizable Verilog; in the modern toolchain that compiler is firtool (from the LLVM-adjacent CIRCT project), a standalone native binary. The Verilog is the hand-off format: FPGA synthesis tools (Vivado) and simulators (Verilator) consume it, not Chisel or FIRRTL.

Diplomacy

An SoC's interconnect has parameters no single component can decide alone: bus widths, address maps, which masters can reach which slaves, interrupt routing. RocketChip solves this with Diplomacy, a negotiation phase that runs before hardware construction: every bus endpoint declares what it needs and supports, the framework propagates and reconciles these along the bus graph, and only then is the actual hardware elaborated with the agreed parameters. Practical consequence: adding a peripheral to the right bus node is enough — widths, address decoding and the device tree entry follow from the negotiation.

Configs and the CDE parameter system

All parameters live in one keyed store (CDE, Context-Dependent Environments): a Field[T] is a typed key, a Config is a layer of key→value bindings, and layers stack — a lookup walks the stack top-down and the first binding wins. “Configuring a design” means composing such layers; a fragment like WithUART overrides exactly the fields it cares about and leaves the rest to the layers below. This is why the assembly order of the stack (next section) is worth understanding: it defines what overrides what.

The device tree

A device tree is a data structure describing the hardware to software: which peripherals exist, at which addresses, with which interrupts and properties. It is compiled from source (DTS) to a binary blob (DTB) that the boot ROM hands to the loaded program. Because the generator emits the device tree from the same elaboration that builds the hardware, software never hardcodes an address — one binary runs on any generated variant of the SoC.

The pipeline at a glance

  1. SOCTLauncher.main parses the arguments, resolves the firtool binary, assembles the Config stack (below), picks the per-target output paths, and dispatches to the target generator (Verilator or Vivado).
  2. Transpiler.evalDesign (implementations linked below) elaborates the design: the top module class is instantiated by reflection with the assembled Parameters, Diplomacy negotiates the interconnect, and the circuit is serialized as FIRRTL (<top>.fir). During this single elaboration the device tree is finalized and the boot ROM is compiled and embedded (see below). RocketChip's elaboration artefacts (.dts, .graphml, .memmap.json, register maps) are written alongside.
  3. Transpiler.emitLowFirrtl / emitVerilog lower the FIRRTL to Verilog into vsrcs/ (details below - the two supported Chisel lineages lower differently).
  4. Target-specific output:
    • Verilator: the emitted Verilog plus SOCTSystem.cmake are consumed by the sim/ CMake project, which verilates the design into a native simulator binary.
    • Vivado: SOCTVivado.prepareForVivado finalizes the block-design graph, patches Vivado interface annotations into the top-level Verilog ports, and emits init.tcl, the block-design TCL, timing constraints and per-component XDC files; optionally Vivado is launched (locally or remotely) to create the project.

How configs shape the build

Everything the generator does is driven by one CDE Parameters stack, assembled in this order (earlier entries win; later ones are fallbacks):

  1. Launcher-argument fragments (from SOCTConfig.apply): WithPeripheryClockSpeed(--periphery-freq-mhz), plus WithSingleBusClockSpeed(--core-freq-mhz), WithFastPnR (--fast-pnr) and RocketChip's WithRV32 (--xlen 32) when requested.
  2. Extra config fragments from --with-config (repeatable) - instantiated by reflection like the base config and inserted here, so they override the base config but not the launcher arguments; among each other, the leftmost occurrence wins. This is how optional features are switched on without editing a config class, e.g. --with-config soct.WithVideoStream.
  3. Your base config - the class named by --config, instantiated by reflection (default soct.RocketB1; typos get a closest-match suggestion). This is where you pick cores (Rocket/BOOM/Gemmini), cache sizes, and any custom fragments.
  4. Launcher-injected context: WithSOCTConfig and WithSOCTPaths expose the resolved arguments and output paths to elaboration-time code (e.g. the boot ROM builder).
  5. Target base configs (appended as fallbacks, so your config overrides them):
    • Vivado: the memory layout from genMemConfig (WithSingleMemLayout/WithMultiMemLayout, sized from the board's DDR4 ports and --ext-mem-part), WithXilinxFPGA(--board), and RocketVivadoBaseConfig (block-design builder, JTAG debug, SD card, UART, full reset scheme).
    • Verilator: RocketSimBaseConfig (HTIF-style simulation setup, synchronous reset scheme, no slave port).

At elaboration time, Fields from this stack gate what gets built: HasUART/HasSDCardPMOD decide whether the UART and SD-card controllers (and their device-tree nodes and PLIC interrupts) exist; XilinxFPGAKey/BdBuilderKey select the board definition and collect the block design; PeripheryClockDomain (a Freq) sets the periphery clock; RegisteredMems/ExtMem define the memory channels; BootROMLocated binds the SOCT boot ROM described next.

Feature fragments that change the design substantially extend SOCTFeatureConfig, which contributes a name suffix: the system name becomes <config>-<xlen>[-<suffix>…] (e.g. RocketB1-64-video for WithVideoStream), and since output paths derive from that name, a feature's artifacts land in their own workspace/ directory instead of overwriting the plain config's.

How the boot ROM is built (single elaboration, lazy contents)

SOCTSystem replaces RocketChip's static boot ROM with SOCTBootROM.attach, whose ROM contents are computed on demand during elaboration - at the moment Diplomacy instantiates the TLROM, the subsystem's device tree is already final. The contents function then:

  1. writes the generated device tree source (<top>.dts),
  2. generates SOCTSystem.cmake (SOCTSystemGenerator extracts the ISA string and CPU count from the DTS and records all paths, ABI and compile definitions),
  3. configures and builds the boot ROM CMake target in binaries/ with Ninja - the target is --bootrom or the target default (sd-boot for Vivado, testchipip-boot for Verilator). The bootrom template compiles the DTS to a DTB (dtc) and embeds it via -DDEVICE_TREE, then objcopys the ELF to a raw bootrom.img,
  4. reads the image bytes back and bakes them into the TLROM.

So the Verilog that comes out of the single elaboration already contains the finished boot ROM - DTB included - and operating systems can read the device tree straight from ROM.

How the lowering works (Transpiler.scala)

There are two Transpiler implementations under src/main/scala-unmanaged/; build.sbt selects one source directory based on the Chisel lineage (SOCT_CHISEL_VERSION):

Modern Chisel (org.chipsalliance, the default) - scala-unmanaged/chisel

  • evalDesign: ChiselStage.elaborate produces the FIRRTL circuit, serialized to <top>.fir.
  • emitLowFirrtl: a no-op - all lowering is delegated to firtool.
  • emitVerilog: invokes the standalone firtool binary on the .fir file with --split-verilog into vsrcs/ (or, with --single-verilog-file, a single .sv with verification layers disabled), --lowering-options=disallowPortDeclSharing (plus locationInfoStyle=none unless --include-location-info), -O=release, and any user --firtool-args.
  • The firtool binary itself comes from --firtool-path, or is resolved by version through firtoolresolver (the version shipped with Chisel, floor 1.75.0; override with --firtool-version).

Legacy Chisel 3 (edu.berkeley.cs, 3.6.1) - scala-unmanaged/chisel3

  • evalDesign: runs the Elaborate and Convert phases manually, writing the FIRRTL circuit plus the annotation file (<top>.anno.json) that later stages need.
  • emitLowFirrtl: runs the Scala FirrtlStage with -E=low-opt, producing optimized low FIRRTL (<top>.opt.lo.fir).
  • emitVerilog: runs FirrtlStage again with -E=verilog --start-from=low-opt and moves the emitted .v files (including blackbox-inline collateral) into vsrcs/.
  • This annotation-driven path is required if your project depends on Transform/FirrtlAnnotation-based concepts, which modern Chisel + firtool do not support.
Tip: to overwrite the Chisel version used by a submodule with its own build.sbt, wrap it in a freshProject call in the root build.sbt.
