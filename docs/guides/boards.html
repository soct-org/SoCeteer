Adding an FPGA Board

A board is one Scala object: it names the part, provides the clock and reset ports, and declares the memory, UART and PMOD resources the systems can use. Everything else (block design, constraints, capacities) is derived from it.

Background

What a “board” means to the tools. Readers who have brought up an FPGA board before can skip ahead.

Part vs. board

The FPGA part is the silicon device; the board is the PCB it is soldered to — with oscillators, buttons, memory sockets and connectors wired to specific package pins. Synthesis only needs the part; everything that touches the outside world needs the board: which pins carry the clock, what voltage a bank runs at, what is attached where. That knowledge exists nowhere in the HDL — it has to be supplied, either from Vivado's board files or as explicit constraints.

Vivado board files and the board flow

For supported boards, Vivado ships board files: XML describing every interface the PCB offers (clocks, DDR4, LEDs), the package pin and electrical standard of every signal, and presets that configure complex IP (like the DDR4 controller) for what is physically on the board. The board flow is Vivado consuming these files — connect an IP to a board interface and pin locations, standards and presets are applied automatically. It is convenient and correct, but only for exactly the hardware the files describe; deviating from it (a different DIMM, an unlisted interface) means placing pins and configuring IP explicitly.

Pins are electrical, not just logical

A package pin assignment always carries an electrical standard (IOSTANDARD, e.g. LVCMOS33 for a 3.3 V single-ended pin, or DIFF_SSTL12 for a 1.2 V differential pair like a DDR4 system clock). The standard is a property of the PCB — wrong values can fail the design-rule checks at best and stress hardware at worst, which is why board definitions record pin location and standard together, taken verbatim from the board files.

The shape of a board definition

Extend FPGA and register the object in FPGARegistry so --board can find it. A board declares four kinds of things: its identity (the FPGA part number and the Vivado board part), its clock and reset inputs, its memory sockets and UART interfaces, and its PMOD pin mapping. The ZCU104 is the reference implementation.

Where the data comes from

Everything a board definition declares exists in the Vivado board files — don't guess:

  • <vivado>/data/xhub/boards/XilinxBoardStore/boards/Xilinx/<board>/<rev>/board.xml — interface names (the values for portName), component wiring.
  • part0_pins.xml (same directory) — every package pin with its IOSTANDARD: clock pairs, reset buttons, PMOD pins, the full DDR4 pin map.
  • preset.xml — the DDR4 preset (memory part, clock periods).

Clock and reset pin locations

Normally Vivado places a board's clock and reset pins automatically, using the board files. The custom DDR4 interface deliberately steps outside that mechanism (see FPGA Memory & Custom DDR4), and then SoCeteer has to place those pins itself — so a board that wants to support it must record the physical package pins of its system clock pair and its reset button in the definition. The ZCU104 declares them right where it creates its clock and reset ports. If a design needs the pins and the board doesn't declare them, generation stops with an error naming the missing field.

Memory ports

Each DDR4 socket is described by a DDR4PortParams: at minimum the DIMM the board files preset (which defines the default capacity), and optionally the pin map and clock timing that allow a different DIMM to be selected with --ext-mem-part. The full story — part registry, board flow vs. custom interface, what fails when data is missing — is in FPGA Memory & Custom DDR4.

Checklist

  1. Create the board object in soct.system.vivado.fpga with the data above.
  2. Register it in FPGARegistry (uppercase key = the --board value).
  3. If the DIMM is user-replaceable, add unknown parts to PartRegistry (or rely on the Micron naming fallback).
  4. Generate with --target vivado --board MYBOARD and let validate_bd_design and the DRCs judge the result.
